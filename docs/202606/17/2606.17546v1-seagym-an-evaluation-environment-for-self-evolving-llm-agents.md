## 摘要
自我进化的基于LLM的智能体主要通过改变其智能体框架来改进：即基础模型周围的结构化执行层，包括提示、记忆、工具、中间件、运行时状态以及模型-工具交互循环。现有评估通常将此过程简化为孤立的任务分数或单一序列曲线，掩盖了更新是否产生可复用的改进、是否过拟合近期任务、是否增加成本或损害旧有行为。我们引入SEAGym，一个用于测量智能体框架更新在训练、验证、测试、重放和成本记录上的评估环境。SEAGym将兼容Harbor的基准测试转化为动态自我进化任务源，包含训练批次、冻结的更新验证、留出的ID和OOD迁移视图、重放诊断以及保存的快照和度量记录。在Terminal-Bench 2.0和HLE上实例化SEAGym，我们在共享epoch/批次协议下比较了ACE、TF-GRPO和AHE。结果表明，这些评估视图提供了关于进化过程的互补信号：频繁更新可能无法改善留出性能，有用的中间快照可能随后崩溃，而源多样性和模型后端可能影响框架可靠性。

## Abstract
Self-evolving LLM-based agents improve mainly by changing their agent harness: the structured execution layer around a base model, including prompts, memory, tools, middleware, runtime state, and the model-tool interaction loop. Existing evaluations often reduce this process to isolated task scores or a single sequential curve, obscuring whether an update produces reusable improvement, overfits recent tasks, increases cost, or harms older behavior. We introduce SEAGym, an evaluation environment for measuring agent harness updates across training, validation, test, replay, and cost records. SEAGym turns Harbor-compatible benchmarks into dynamic self-evolution task sources with train batches, frozen update-validation, held-out ID and OOD transfer views, replay diagnostics, and saved snapshot and metric records. Instantiating SEAGym on Terminal-Bench 2.0 and HLE, we compare ACE, TF-GRPO, and AHE under a shared epoch/batch protocol. The results show that these evaluation views provide complementary signals about the evolution process: frequent updates may fail to improve held-out performance, useful intermediate snapshots may collapse later, and source diversity and model backend can affect harness reliability.

## 6. 论文的主要结论与发现

1. **不同方法产生不同的更新动态**：
   - AHE编辑框架本身，可同时改进验证、ID和OOD（验证+17.1，ID+9.1，OOD+6.3），但更大的编辑范围带来可靠性风险。
   - ACE积累可复用的技能知识，提升有限（验证+2.9，ID+3.6，OOD+2.5），对运行时行为失败控制力弱。
   - TF-GRPO从分组rollout中快速强化源分布行为，验证增益大（+17.1），但OOD下降（-2.5）且成本最高。

2. **训练回放揭示框架不稳定**：AHE的中间快照E16在80个训练任务中仅成功6个，原因是**消息契约违规**（运行时中间件注入不兼容消息）。最终快照恢复（43/80），但净增益仅9个任务（修复13个初始失败，遗忘4个初始成功）。这证明初始—最终比较会掩盖危险的中间回归。

3. **批次大小非单调**：批次20是唯一产生大增益的设置；批次10和80均导致验证和ID下降。小批次更新频繁且证据窄，易积累运行时风险；大批次证据过载，稀释每个任务的分析，且修复机会少。

4. **源多样性影响恢复能力**：仅HLE源训练在E12达到有用中间快照（ID+7.3），但最终快照崩溃（验证0%），因为更新集中在验证子系统，最终消息契约失效导致所有任务失败。混合源提供更多样的失败证据（工具、环境、文件等），有助于从运行时契约错误中恢复。

5. **跨模型迁移不对称**：相同模型后端的ID增益为正，但跨后端时增益不稳定甚至为负（如DeepSeek进化框架提高GLM的ID，但降低GPT-5.4的ID）。OOD增益几乎为负或中性，表明更新拟合了训练后端的失败模式，但未泛化到其他后端或分布。

6. **更新验证增益不等于迁移增益**：TF-GRPO验证增益大但OOD损失；AHE的验证增益与ID/OOD正相关，但中间可能崩溃；ACE增益小但相对稳定。单一曲线不足以评估自我进化。

## 7. 优点：方法或实验设计上的亮点

- **系统性强**：将自我进化形式化为MDP评估过程，明确区分智能体快照（基础模型+框架状态）和更新规则，支持多种调度方式。
- **模块化设计**：通过rollout/update接口集成不同方法，保留原生更新规则，无需重写基准测试。
- **多视图评估**：提供更新验证、ID迁移、OOD迁移、重放诊断、成本记录等互补信号，比单一最终分数更全面。
- **诊断能力**：保存轨迹、更新工件、快照差异，支持离线分析为何改进/回归/恢复，例如发现执行路径不稳定、消息契约错误等深层原因。
- **实验设计细致**：包含批次大小、源多样性、跨模型转移等消融，并通过大量案例研究（附录C）解释更新机制，结论有扎实证据。
- **可扩展性**：基于Harbor框架，可将更多基准转换为自我进化任务源，只需增加任务适配器和薄包装器。

## 8. 不足与局限

- **领域覆盖有限**：仅使用Terminal-Bench 2.0和HLE（文本型），未涵盖网页/桌面交互、长时软件工程、数据分析工作流、多智能体协作、连续在线流等场景。未来需扩展。
- **仅研究框架级进化**：不涉及模型权重更新（如在线RL微调）。论文指出环境可扩展，但当前实验未覆盖。
- **成本/覆盖权衡**：保存快照并在多个视图上重新评估带来额外计算开销。未讨论自适应快照选择或预算感知评估。
- **资源细节缺失**：未报告GPU型号、数量、显存、并行策略，影响重现性和效率比较。
- **OOD与跨模型实验有限**：仅各一个目标域和三个模型，未涵盖更广泛的领域/模型组合。更新对后端和分布的依赖可能随更多源-目标对变化。
- **重放诊断依赖固定任务集**：训练集回放可能过拟合特定任务，未考虑任务随时间变化或新任务涌现。
- **安全与隐私**：未深入讨论评估框架在涉及真实用户数据或专有仓库时的合规性和匿名化要求（仅在伦理声明中简要提及）。
