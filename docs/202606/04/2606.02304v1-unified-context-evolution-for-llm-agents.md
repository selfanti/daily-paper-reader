## 摘要
基于LLM的智能体能够通过结合推理与环境反馈解决多步交互任务，但每个回合都从相同的固定上下文开始，且任务结束后沿途发现的任何有用策略都会丢失。现有方法要么将学习限制在当前任务内，要么将所有经验汇集到一个无类型的单一存储中，而不区分知识类型、通过使用追踪质量、或平衡库中仍缺失的内容。我们提出统一上下文演化（UCE），一种无梯度框架，将智能体经验外部化到可演化的类型化可演化上下文单元（ECU）库中。UCE将经验分解为四种互补类型（记忆、策略、工作流和技能），每种类型从轨迹中在类型特定条件下生成，在决策时检索，通过重复使用结果进行评分，并在不再有价值时进行剪枝。调度模块将每个周期的生成预算分配给库中最弱的类型。在两个交互基准测试中，UCE将ALFWorld成功率从75.4%提升至96.3%，WebShop任务得分从45.1%提升至61.3%，且积累的库无需重新训练即可迁移到其他智能体骨干网络。

## Abstract
LLM-based agents can solve multi-step interactive tasks by combining reasoning with environment feedback, yet each episode starts from the same fixed context and any useful strategy discovered along the way is lost once the task ends. Existing approaches either limit learning to the current task or pool all experience into a single untyped store, without distinguishing knowledge types, tracking quality through use, or balancing what the library still lacks. We introduce Unified Context Evolution (UCE), a gradient-free framework that externalizes agent experience into an evolving library of typed Evolvable Context Units (ECUs). UCE decomposes experience into four complementary types (Memory, Strategy, Workflow, and Skill), each generated from trajectories under type-specific conditions, retrieved at decision time, scored through repeated usage outcomes, and pruned when no longer valuable. A scheduling module allocates each cycle's generation budget toward the types where the library is weakest. Across two interactive benchmarks, UCE raises ALFWorld success from 75.4% to 96.3% and WebShop task score from 45.1% to 61.3%, and the accumulated library transfers to alternative actor backbones without retraining.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：当前基于LLM的智能体（Agent）在执行多步交互任务时，每个回合（Episode）都从一个固定上下文（如系统指令、少量示例）开始。智能体在任务执行过程中学习到或发现的任何有用策略、环境机制或工作流程，都会在任务结束后丢失，无法跨任务积累和复用。这导致了严重的效率低下和知识遗忘问题。
- **研究背景**：
    - **现有解决方案的不足**：
        - **强化学习 (RL)**：效果好，但需要访问模型权重和大量训练资源，不适用于闭源模型。
        - **任务内适应 (Within-Task)**：如 Reflexion 在单任务内进行反思，但改进无法跨任务迁移。
        - **跨任务知识库 (Cross-Task)**：如 ExpeL 尝试积累知识，但大多将所有经验混入一个无类型的知识池，未能区分知识类型（如环境事实、决策规则、工作流程），缺乏基于使用结果的质量追踪，也没有达成知识库内容类型的平衡。
- **整体含义**：文章旨在解决“LLM智能体如何高效地、结构化地、动态地积累和复用跨任务经验”这一核心问题。其核心理念是**让上下文本身成为学习和演化的载体**，从而在不更新模型权重的情况下，实现智能体能力的持续提升。

## 2. 方法论

- **核心思想**：提出 **统一上下文演化（Unified Context Evolution, UCE）** 框架。该框架将智能体的经验**外部化**到一个动态演化的**类型化可演化上下文单元 (Evolvable Context Units, ECUs) 库**中。整个学习过程是**无梯度**的，完全通过管理这个知识库来实现。
- **关键技术细节**：
    1.  **Evolvable Context Units (ECUs)**：知识库的基本单元，每个 ECU 是一个五元组 `(类型, 任务类型, 检索条件, 内容, 适应度评分)`。ECU 被设计为四种功能互补的类型：
        - **记忆 (Memory)**：捕获单一任务中发现的**环境机制**（如“洗手池无需打开水龙头即可清洁物品”）。
        - **策略 (Strategy)**：编码**条件决策规则**（如“如果”放入失败，“则”打开容器重试）。
        - **工作流 (Workflow)**：描述任务类型的**标准执行序列**（如“找到目标 -> 转换它 -> 放置它”）。
        - **技能 (Skill)**：指定**可跨任务类型重用的子操作**（如“系统搜索容器”的方法）。
    2.  **检索机制**：在执行任务前，根据**语义相似度、任务类型匹配度、适应度评分**对库中的ECU进行排序，然后**按类型独立**选出Top-K个ECU，注入到智能体的提示词中。这种做法避免了单一类型知识淹没其他类型。
    3.  **适应度评分与生命周期**：每个ECU都有一个基于使用结果的**Laplace平滑适应度分数** `f_e = (s_e + 1) / (u_e + 2)`。`s_e`是累积奖励，`u_e`是使用次数。新ECU的初始分数为0.5。当ECU使用次数足够且分数低于阈值时，会被**剪枝**。这实现了基于实证效果的自动质量控制。
    4.  **知识收益调度 (Knowledge Yield Scheduling, KYS)**：一个三层调度模块，决定每个演化周期为每种ECU类型分配多少生成预算：
        - **第一层：可行性过滤 (Feasibility Filtering)**：判断该类型是否还有知识缺口（如还有任务类型未被覆盖）。
        - **第二层：需求评估 (Demand Assessment)**：根据覆盖率和质量缺口计算需求量。
        - **第三层：收益追踪 (Yield Tracking)**：基于历史收益（上一个周期的成功率变化），使用指数移动平均（EMA）计算预期的回报率。
        - **最终决策**：综合需求和预期回报，为每种可行的类型分配生成预算，确保资源被投入到最薄弱的知识领域。
    5.  **演化周期 (Evolution Cycle)**：每个周期包含五个阶段：
        1.  **评价 (Evaluate)**：在**held-out**评估集上运行智能体，记录性能，**不更新适应度分数**。
        2.  **收集 (Collect)**：在**不相交**的收集集上运行智能体，记录轨迹和奖励，**仅在此阶段更新适应度分数**。
        3.  **调度 (Schedule)**：KYS模块计算每种ECU类型的生成预算。
        4.  **生成 (Generate)**：根据预算，使用类型特定的生成器从收集到的轨迹中蒸馏出新的ECU候选者，进行去重后加入库中。
        5.  **清理 (Cleanup)**：剪枝掉低质量和冗余的ECU，形成下一代库。

## 3. 实验设计

- **数据集与场景**：
    - **ALFWorld**：一个基于文本的家居环境任务，包含6种任务类型（如取放、清洁、加热、冷却等），共134个任务。
    - **WebShop**：一个基于文本的在线购物环境，任务来自Amazon产品页面，包含5种子类型（如美妆、服装、电子产品、食品、家居），共100个任务。
- **基准 (Benchmark)**：在这两个标准交互式基准上进行评估。
- **对比方法**：
    - **跨范式基线**：ExpeL (经验池方法), Reflexion (任务内迭代反思方法)。
    - **不同Actor骨干网络**：在不使用UCE和使用UCE的情况下，测试了四种智能体推理格式：
        - **ReAct** (默认基准)
        - **NoThinking** (直接输出动作)
        - **Plan-and-Act** (首步规划，后续直接动作)
        - **ReflAct** (每步进行状态反思)
- **公平性保障**：所有对比方法都使用相同的**Actor模型（gpt-4.1-mini）**、**生成器模型（gpt-5.2）**、**任务集合**和**步数限制**。实验采用了**分层交叉验证**，确保评估的客观性。

## 4. 资源与算力

- **明确说明**：论文明确指出所有实验均通过**官方OpenAI API**进行，**不需要本地GPU**。这是由于其“无梯度”框架的特性，模型权重不更新，只管理知识库。

## 6. 论文的主要结论与发现

1.  **显著性能提升**：UCE在两个基准上均取得了大幅领先。在ALFWorld上，成功率从75.4%提升至**96.3%** (+20.9个百分点)；在WebShop上，任务得分从45.1%提升至**61.3%** (+16.2个百分点)，购买成功率从30.0%提升至**42.0%**。
2.  **知识库的可迁移性**：在UCE框架下学习到的ECU库本身具有通用性，可以直接与不同推理格式的Actor骨干（如NoThinking, Plan-and-Act, ReflAct）结合，并带来一致的性能提升，无需重新训练。
3.  **不同类型ECU的价值**：消融实验表明，在ALFWorld上，**工作流（Workflow）** 类型贡献了主要增益；而在更复杂的WebShop上，**全类型组合**的效果超过了任何单一类型，说明复杂场景需要多种知识协同工作。
4.  **知识库动态演化的有效性**：通过知识收益调度（KYS）和生命周期管理（评分与剪枝），UCE的库能够在少数几个周期内快速收敛到高性能水平，并能将有限的生成资源分配到最需要的知识领域。

## 7. 优点

1.  **无梯度、轻量级、可迁移**：不修改模型权重，使方法可以应用于闭源API模型，且库本身可以跨骨干网络迁移，实用性极强。
2.  **结构化、类型化知识设计**：将经验分解为四种互补的知识类型，避免了单一知识池的混乱，并通过不同类型的协同作用解决复杂问题。设计精巧。
3.  **基于实证的生命周期管理**：通过使用次数和结果（适应度评分）来客观衡量ECU的质量，并据此进行检索（质量偏置）和剪枝，实现了数据驱动的自动演进。这是对简单“一次生成，永久使用”方法的重大改进。
4.  **动态的生成预算调度（KYS）**：能根据库当前的状态和历史收益，智能地将资源分配给最薄弱的知识类型，这是一种高效且自适应的学习策略。
5.  **严谨的实验设计**：分离的评估/收集流程、多骨干测试、统计显著性检验，都提升了结果的可靠性和说服力。

## 8. 不足与局限

1.  **基准与场景局限性**：实验仅限于两个基于文本的交互环境（ALFWorld, WebShop），它们具有离散动作空间和相对规则的结构。作者承认未在**连续动作、多模态观测**（如AndroidWorld, VisualWebArena）或更长规划范围的场景上进行验证，因此结论的泛化能力有待考察。
2.  **模型依赖**：实验仅使用单一模型对（`gpt-4.1-mini` + `gpt-5.2`）。作者承认未能解耦性能增益是来自框架设计还是其生成器（`gpt-5.2`）强大的知识蒸馏能力。
3.  **任务规模问题**：两个任务池都相对较小（134 和 100）。在**更大规模的任务集**上，KYS的信号是否依然可靠，库是否会继续增长或最终退化，是未解决的开放问题。
4.  **类型设计固定**：当前的四种ECU类型是手工设计的。作者指出，**自动发现新类型**以适应新领域是未来的方向。这意味着当前方法在不同领域可能需要重新设计类型体系。
5.  **潜在的混杂因素**：尽管实验设计公平，但“无类型”基线（如ExpeL）的库中可能隐含了与UCE相似的分类，只是未显式管理，因此UCE的部分优势可能来自于更清晰的结构化管理本身。一个更深入的消融研究（例如，将UCE的四种类型提示词与一个包含相同数量但无类型区分的提示词进行对比）可能能更好地分离出类型设计的贡献。
