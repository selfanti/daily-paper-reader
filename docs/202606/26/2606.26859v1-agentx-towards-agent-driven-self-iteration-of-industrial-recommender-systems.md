## 摘要
推荐算法的迭代正从一种依赖工程师的手工艺过程，转向工业化的研究循环，但这一转型仍受到结构性执行瓶颈的阻碍：从想法到上线的周期仍然依赖人类工程师生成假设、修改生产代码、启动A/B实验以及归因线上结果。因此，创新仅随人力线性增长，而非在证据、计算和积累的实验知识中复合增长。我们提出AgentX，一个部署于生产环境的多智能体系统，它从根本上重构了这一生产函数。AgentX作为一个自我进化的开发引擎运作：它以手工流程无法企及的规模和速度自主生成、实施、评估推荐实验，并从中学习。该系统以闭环方式编排四个紧密耦合的阶段。头脑风暴智能体综合历史实验、系统架构、数据分析和外部研究的证据，生成排序后的可执行提案。开发智能体通过基于仓库的生成和多维可靠性验证，将每个提案转化为生产就绪的代码。评估智能体通过带护栏否决的A/B判断进行安全在线部署，将成功与失败都转化为结构化的知识资产。随后，一个驾驭进化层（SGPO）将执行轨迹提炼为语义梯度更新，持续强化智能体自身——使得系统不仅自动化，而且能够自我改进。

## Abstract
Recommendation algorithm iteration is moving from an artisanal, engineer-bound process toward an industrialized research loop, but this transition remains blocked by a structural execution bottleneck: the idea-to-launch cycle still depends on human engineers to generate hypotheses, modify production code, launch A/B experiments, and attribute online results. Innovation therefore scales linearly with headcount rather than compounding with evidence, compute, and accumulated experimental knowledge. We present AgentX, a production-deployed multi-agent system that fundamentally restructures this production function. AgentX operates as a self-evolving development engine: it autonomously generates, implements, evaluates, and learns from recommendation experiments at a scale and pace that no manual workflow can sustain.   The system orchestrates four tightly coupled stages in a closed loop. A Brainstorm Agent synthesizes evidence from historical experiments, system architecture, data analysis, and external research into ranked, executable proposals. A Developing Agent translates each proposal into production-ready code through repository-grounded generation and multi-dimensional reliability verification. An Evaluation Agent conducts safe online rollout with guardrail-vetoed A/B judgment, converting both successes and failures into structured knowledge assets. A Harness Evolution layer (SGPO) then distills execution trajectories into semantic-gradient updates that continuously sharpen the agents themselves -- making the system not merely automated, but self-improving.

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- **闭环迭代**：AgentX构建了一个完整的闭环：**想法生成 → 代码实现 → 在线评估 → 经验积累 → 智能体自身进化**。将人类工程师从重复执行中解放出来，转向更高价值的问题定义和系统设计。
- **多智能体协作**：系统包含四个紧密耦合的智能体，分别负责不同阶段，并通过共享的数据层（知识库、实验轨迹）和监控平台协同工作。
- **自我进化**：通过语义梯度提示优化（SGPO）和模型研究探索管道，系统能基于历史执行轨迹持续优化自身提示、规则和工作流，实现长期能力增长。

### 关键技术细节

#### 4个智能体
1.  **头脑风暴智能体（Brainstorm Agent）**：
    - 将模糊的目标意图转化为一组可执行的、基于证据的实验提案。
    - 通过**边界探索**（限制搜索空间）、**证据加权生成**（综合实验知识库、系统知识库、数据分析、模型研究四类证据）和**验证/具体化**（人工审批、边界检查）来保证提案质量。
    - 每个提案附带因果链、预期观测值和成熟度状态（Ready-to-implement / Probe-first / Moonshot）。

2.  **开发智能体（Developing Agent）**：
    - 并行处理两条轨道：**在线策略开发**（生产代码修改）和**离线模型开发**（模型架构实验）。
    - **在线策略开发**：基于仓库知识的代码生成 + 验证导向的实现循环。使用模式工具箱（特征模式查询、DSL检查器、C++语法检查等）防止属性幻觉、DSL误用等可靠性问题。质量评分涵盖8个维度（如属性幻觉、正确性循环次数、人工干预等）。
    - **离线模型开发**：采用“策略 → 代码 ↔ 验证 → 执行 → 最终评审”的流水线。严格保证**可证伪归因**：每个实验必须声明因果机制和预期观测值，最终由独立专家智能体投票（超2/3多数）确认，并且基于原始训练日志的确定性指标提取（正则表达式），确保结论可信。对于平台故障（GPU OOM、集群停滞等）有分类和重试策略。

3.  **评估智能体（Evaluation Agent）**：
    - 管理安全的在线部署（流量分配、灰度发布、平衡检查）和A/B判断。
    - 使用CUPED等方差缩减方法，并采用**带护栏否决的A/B判决**：护栏是业务范围性的、复合经济交换指标，且阈值设计为“注意力信号”而非绝对阻断，减少误判。
    - 将负结果（失败、无结论）资产化：记录失败根因并索引到知识库，供未来头脑风暴和探索避开相同路径。

4.  **驾驭进化层（Harness Evolution）—— SGPO**：
    - **SGPO-I**：基于会话轨迹（对话历史），提取失败诊断作为语义梯度，然后由精炼智能体修改目标智能体的提示（指令、验证规则、输出合同、工具使用纪律），并通过**成对回放**（同一任务在老/新提示下的得分对比）决定是否采纳。
    - **SGPO-II**：编码阶段的进化，基于历史合并请求（MR）构造回放案例，评估维度包括语义正确性、需求覆盖、文件覆盖、安全默认、代码风格。两个版本都是在固定其他组件（基础模型、工具接口、顶层编排）的前提下更新单个子智能体，保证可审计性和安全。

#### 模型研究探索管道
- 基于结构化的论文知识库（事实存储：声明、架构、关系），执行三阶段循环：**论文复现 → 模块消融 → 跨论文组合**。每阶段从更大的论文池（1312篇）中逐步筛选，通过记忆机制（论文级/模块级剪枝、饱和检测）和双门控经验验证（阈值门+对抗审查门）确保探索高效且可信。

## 3. 实验设计

### 部署场景

- **Benchmark**：与单个算法工程师（传统手工迭代）进行直接对比。

### 对比方法和指标
- **对比方法**：无直接对比的学术方法，主要是与人类工程师的生产率（单周并发实验数、累计上线结果数、业务价值（用户活跃时间增量））进行对比。

    - **漏斗转化率**：想法数 → 通过验证数 → 上线数 → 正面评估数（Launchable Results, LR）。

    - **AgentX vs. 工程师**：每位worker每周并发的实验数、上线结果数、用户活跃时间贡献、单想法的上线转化率。

    - 公开数据集实验仅以AUC为唯一指标，缺乏其他离线指标（如logloss、NDCG）及统计显著性报告。
    - 未与现有AutoML/agent基线（如AutoML-Agent, AIDE, MLEvolve等）进行对比，论文中虽然讨论了相关工作，但实验部分未实现这些方法进行公平比较（仅与人类工程师对比）。虽然作者声称这是工业系统的独特优势（真实的在线反馈和规模），但缺乏与学术SOTA的直接对比影响了严谨性。
    - 生产环境样本量（3周、3个worker）较小，且结果高度依赖于快手的具体业务场景（如主Feed和生活服务），通用性有待验证。
    - 未进行大规模超参数或随机种子实验，所有复现结果基于单次运行？论文未说明训练种子策略。

## 6. 论文的主要结论与发现

1.  **AgentX实现了可自我进化的工业推荐系统迭代闭环**。系统能自主完成想法生成、代码实现、安全上线、结果反馈和智能体升级的全流程，且每次实验的经验（包括失败）都能积累为结构化知识，用于后续迭代。
2.  **显著提升了迭代吞吐量和业务价值**：与单个工程师相比，每worker每周并发实验数提升8倍，上线结果数提升13.8倍，累计用户活跃时间增长贡献提升3.7倍。
3.  **主要的执行瓶颈来自基础设施和平台约束**（如AB平台参数冲突、数据特征缺失），而非智能体推理能力不足。这表明未来优化的方向是构建上游冲突检测器，而非一味提升智能体。
4.  **离线模型探索管道能有效复现论文、消融模块并进行跨论文组合**，且工业场景验证了迁移性。模块组合如MHFT+VQ_Code+MVSF在公开和工业数据集上均表现出色。
5.  **语义梯度提示优化（SGPO）是一种安全、可审计的智能体进化方法**，通过成对回放确保更新不会退化系统性能。进化过程可能非单调，但守门机制（成对回放）保证了生产稳定性。

## 7. 优点

- **现实导向**：直接使用真实的在线A/B反馈作为奖励信号，而非离线指标模拟，这使系统优化目标与业务价值完全对齐。
- **完整的闭环设计**：不仅实现自动化，还包含自我进化机制，使系统能长期复合增长，这是现有ML agent系统欠缺的。
- **细粒度的可靠性工程**：在开发阶段设计了多维度质量评分、属性查询工具、DSL检查器、成对回放等机制，显著降低了生成代码在工业生产中引入缺陷的风险。
- **负结果资产化**：将失败的实验系统化记录为可重用的知识（根因、业务上下文），避免后续反复重复错误方向，提高了整体效率。
- **开源的实验框架（至少技术报告公开）**：附件中提供了详细的提示格式，便于社区复现和扩展。

## 8. 不足与局限

- **实验覆盖的局限性**：
    - 公开数据集实验仅用AUC一个指标，且未报告方差或显著性，削弱了结论的可信度。
    - 生产实验仅在一个平台（快手）、两种场景下进行三周，样本量有限，长期稳定性（如系统进化是否会过拟合到当前流量模式）未讨论。
    - 未与其他多智能体ML系统（如Data Interpreter, MLEvolve）进行直接比较。由于其他系统不直接使用在线A/B反馈，对比可能有难度，但论文可以构建离线模拟实验来展示AgentX的优越性。
- **算力消耗不透明**：未报告训练和部署AgentX所需的GPU/CPU资源，无法评估其成本效益比。与人类工程师对比时，未计算AgentX消耗的计算成本（包括LLM推理、训练、实验运行），可能低估了能耗和费用。
- **泛化性风险**：AgentX的设计高度依赖快手内部的知识库（系统KB、实验KB、DSL等），将其迁移到其他推荐系统需要重新构建这些知识库，成本较高。
- **提示工程依赖**：SGPO虽然自动进化提示，但初始提示（见附录）仍需手动设计，且当前只更新单个子智能体，进化速度受限。评估器本身也是LLM，其诊断质量可能存在偏差，文中也承认了“LLM-as-judge misdiagnosis”的风险，虽然成对回放能阻挡退化，但不能保证进化的最优路径。
- **伦理/业务风险**：论文未讨论如果系统持续优化用户参与度（如活跃时间），可能带来的成瘾性或用户福祉问题。自动化推荐策略迭代可能会放大现有推荐偏差。
