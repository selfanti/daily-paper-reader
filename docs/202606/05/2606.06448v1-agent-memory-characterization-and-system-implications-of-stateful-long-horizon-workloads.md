## 摘要
LLM代理越来越多地部署在长周期任务上，这些任务需要在扩展的交互历史中进行持续推理。大规模实现这一点要求代理跨会话持久地存储、检索和更新自己的记忆。一个丰富的代理记忆系统生态系统已经出现，涵盖平面检索、LLM介导的提取、合并事实存储以及代理控制流。然而，它们的系统级行为仍未得到表征。我们首次对代理记忆进行了系统表征。首先，我们引入了一个面向系统的分类法，沿着四个轴对代理记忆系统进行分类。其次，我们构建了一个阶段感知的性能分析工具，将成本归因于构建、检索和生成。第三，我们在两个基准套件上表征了十个代表性系统，揭示了设计选择如何在写入和读取路径上转移成本。最后，我们得出了10条系统建议，涵盖构建调度、能力下限、通过查询量进行摊销、新鲜度-延迟权衡以及集群规模管理。

## Abstract
LLM agents are increasingly deployed on long-horizon tasks requiring sustained reasoning over extended interaction histories. Realizing this at scale requires agents to persistently store, retrieve, and update their own memory across sessions. A rich ecosystem of agent memory systems has emerged spanning flat retrieval, LLM-mediated extraction, consolidating fact stores, and agentic control flows. Yet, their system-level behavior remains uncharacterized. We present the first systems characterization of agent memory. First, we introduce a system-oriented taxonomy classifying agent memory systems along four axes. Second, we build a phase-aware profiling harness attributing cost to construction, retrieval, and generation. Third, we characterize ten representative systems across two benchmark suites, uncovering how design choices shift cost across the write and read paths. Finally, we derive 10 system recommendations covering construction scheduling, capability floors, amortization via query volume, freshness-latency tradeoffs, and fleet-scale management.

### 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM代理在长周期任务（如跨数周的对话、代码编辑、深度研究）中需要维护大量交互状态，现有外部记忆系统（如RAG的变体）虽然旨在解决单次上下文窗口的限制，但其系统级行为（成本、延迟、资源消耗）从未得到系统性的量化和理解。这导致运维人员无法基于成本、延迟和准确性三维度做出合理的系统选择。
- **动机与背景**：从纯参数化知识到RAG再到状态化代理记忆，记忆从被动检索目标变为主动系统组件（具有写入、搜索和维护路径）。现有基准只评估下游准确性，忽略计算成本。本文填补这一空白，首次对代理记忆进行全面的系统级特征分析。

### 2. 方法论：核心思想、关键技术细节

- **核心思想**：提出一个面向系统的四轴分类法，将代理记忆系统划分为四种范式，并构建阶段感知分析工具，将成本归因于构造、检索和生成三个阶段。
- **四轴分类法**：
  - **构造机制**：无（长上下文）、确定性（分块/索引）、LLM介导（提取事实/摘要）、代理控制流（LLM自主决定写操作）。
  - **存储组织**：原始上下文、倒排索引、向量索引、知识图谱、多存储组合。
  - **检索策略**：单次稀疏/密集搜索、多阶段流水线（包含重排、图遍历）、代理化工具调用。
  - **可变性**：仅追加、合并、变异。
- **四种范式**：
  - **I. 长上下文记忆** （如long_context）：不构建外部记忆，直接全历史在上下文中。
  - **II. 平面RAG记忆** （如BM25, embedRAG）：确定性索引，无LLM参与构造，仅追加。
  - **III. 结构增强RAG记忆**：进一步分为仅追加（GraphRAG, HippoRAG v2）和合并型（Mem0, SimpleMem），用LLM作为固定提取器生成结构化记录。
  - **IV. 代理控制流** （如A-Mem, Letta, MIRIX）：LLM控制记忆读写，可变深度调用。
- **阶段感知分析框架**：对每个系统的执行分解为七个阶段（摄入、构造、存储、检索、提示组装、生成、维护），并通过API遥测（调用类型、token数、延迟）和硬件遥测（GPU利用率、功率、HBM带宽）在统一时间轴上归因成本。

### 3. 实验设计与基准

- **基准套件**：
  - **MemoryAgentBench (MAB)**：主要基准，包含4个任务类别（准确检索、测试时学习、长距离理解、选择性遗忘），每个样本以4096-token分块增量输入。中心数据集是LongMemEval_S_\*（约360K token历史，60个查询，共300个查询）。
  - **MemoryArena**：用于新鲜度-延迟权衡实验，包含20个多会话任务，每会话依赖之前会话结果。
- **对比方法**：10个代表系统：long_context, BM25, embedRAG, GraphRAG, HippoRAG v2, Mem0, SimpleMem, A-Mem, Letta, MIRIX。
- **标准化**：统一使用相同的生成LLM和嵌入模型（条件许可时），检索时固定最多10个记忆条目，对系统进行最小提示适配以保证公平，但对原生设计（如并行摄入、跨块合并）保留。

### 6. 主要结论与发现

论文总结为8个洞见（Insights）和10条建议（Recommendations）。核心发现：
- **构造主导生命周期**：对于LLM介导系统，构造能耗超过300次查询的总能耗，能量/正确答案差异超47倍。
- **构造是预填充和嵌入密集型**：约95%的构造token是嵌入或LLM提示（预填充），解码占比很小，导致与延迟敏感的QA服务产生资源竞争。
- **构造LLM选择受算法约束**：有严格输出契约的系统（如MIRIX）存在最小LLM能力地板，低于则存储损坏；无硬契约系统可平滑降级。
- **无系统同时优化三维**：BM25构造快、准确率高但查询慢；Mem0查询极快但构造慢、准确率低。系统选择需匹配工作负载特征（查询到达率 vs 连续摄入）。
- **会话间构造产生新鲜度-延迟权衡**：慢构造系统在异步调度下产生记忆陈旧（staleness），每会话构造时间跨度达5个数量级，决定了可行性。
- **每用户内存足迹增长分歧大**：在1M token时差异9倍，但构造token成本增长斜率分歧更大（尤其代理控制系统超线性增长）。存储和token成本需长期规划。
- **服务延迟由算法界定步骤 vs LLM界定步骤决定**：算法界定系统（如BM25）尾延迟窄（p95/p50≈1.3x），LLM界定系统（如GraphRAG, Letta）尾延迟宽（可达5.9x），需要外部迭代上限。

### 7. 优点

- **首次系统级表征**：填补了代理记忆领域系统行为量化的空白。
- **完整分类法**：基于构造、存储、检索、可变性四轴，清晰预测了每种范式的成本形状和适用场景。
- **阶段感知分析工具**：可归因成本至不同阶段，为系统优化提供依据。
- **跨多维度定量比较**：涵盖能耗、延迟、token消耗、存储足迹、调度影响、可扩展性等。
- **提供实操建议**：10条具体建议直接指导部署决策（如构造作为后台吞吐任务、避免与QA共存、为LLM界定步骤设外部上限）。

### 8. 不足与局限

- **单节点局限**：所有实验在单GPU上运行，未涉及分布式/多节点部署，而真实生产环境多为集群。
- **多智能体与多模态未覆盖**：多智能体协调一致性问题、图像/音频记忆的额外挑战不在本工作范围内，但作者指出未来工作方向。
- **标准化难度**：某些系统需要提示适配（如Mem0提取提示更改为ICL-aware），可能引入微小偏差；Letta本地模型因重复工具调用而对容量进行了限制。
- **准确性评估有限**：基准任务集中于信息检索，未充分测试复杂推理、多跳关系、时间推理等——而这些正是结构增强系统设计针对的场景。
- **遗忘策略缺失**：所有系统默认无限累积状态，未探讨遗忘或压缩策略对成本的影响。
- **部署建议依赖具体测量**：部分建议（如构造LLM成本杠杆）基于特定模型和硬件，迁移性需验证。
