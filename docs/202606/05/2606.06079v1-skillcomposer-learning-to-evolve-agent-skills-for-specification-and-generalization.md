## 摘要
智能体技能由可重复使用的策略组成，这些策略指导智能体的推理和行动，在推理时提升模型能力方面展现出巨大潜力。然而，当前的技能构建方法将这一问题视为一次性提取，忽略了一个根本性矛盾：针对特定任务定制的技能无法迁移，而抽象化的技能往往提供的指导不足。我们将这种脆弱性归因于缺乏技能规范与泛化的显式机制。为解决这一差距，我们引入了SkillComposer，这是一个将技能构建分解为三个可学习操作（创建、改进、合并）的框架。通过系统的拒绝采样训练方法，SkillComposer使语言模型能够在推理时自我演化技能，并支持三种部署模式：用于构建通用库的离线模式、用于任务特定优化的在线模式以及结合两者的混合模式。在τ^2-Bench、LiveCodeBench v6和AppWorld上的全面实验表明，SkillComposer持续优于基线方法。我们的SkillComposer-4B在智能体任务上将27B执行器的性能提升了最多+4.5，在代码任务上提升了+3.4，同时能够泛化到训练中未见过的领域和任务类型。分析表明，合并与改进操作解决了正交的质量维度，并且技能组合是一种可迁移的元能力，为技能增强推理提供了一种实用的方案。

## Abstract
Agent skills, which consist of reusable strategies that guide agent reasoning and action, have shown strong potential for improving model capability at inference time. However, current skill construction methods treat the problem as one-shot extraction, overlooking a fundamental tension: a skill tailored to the specific task fails to transfer, while the abstracted skill often provides insufficient guidance. We attribute this fragility to the absence of explicit mechanisms for skill specification and generalization. To address this gap, we introduce SkillComposer, a framework that decomposes skill construction into three learnable operations: create, improve, and merge. Trained via systematic rejection sampling recipe, SkillComposer enables language models to self-evolve skills at inference time and supports three deployment modes: offline for building generalized libraries, online for task-specific refinement, and hybrid for combining both. Comprehensive experiments on $τ^2$-Bench, LiveCodeBench v6, and AppWorld show that SkillComposer consistently outperforms baselines. Our SkillComposer-4B improves a 27B executor by up to +4.5 on agent tasks and +3.4 on code tasks, while generalizing across domains and task types unseen during training. Analysis reveals that merge and improve address orthogonal quality dimensions and that skill composition is a transferable meta-ability, providing a practical recipe for skill-augmented inference.

- **核心问题**：现有智能体技能构建方法（如从成功轨迹中一次性提取）忽视了技能质量中“规范”（task-specific）与“泛化”（task-agnostic）之间的根本矛盾。过于具体的技能无法迁移，过于抽象的技能指导不足，导致性能提升有限甚至负收益。

- **整体含义**：SkillComposer将技能构建视为一个可学习的“技能演化”过程，通过三个基本操作（创建、改进、合并）让语言模型在推理时自主演化技能，从而同时实现规范与泛化。

### 2. 论文提出的方法论
- **核心思想**：将技能操作分解为三个可学习的能力——Skill Create（从原始轨迹提取可复用技能）、Skill Improve（基于新执行经验细化技能，驱动规范）、Skill Merge（合并相似技能为更通用的技能，驱动泛化）。这三个操作共同定义技能生命周期：从提取到跨任务抽象再到迭代细化。
- **关键技术细节**：
  - 技能定义为三元组：名称`n`、描述`d`、主体`b`。
  - **Skill Merge** 使用多视图相似度度量：`sim(s1,s2) = 1/3 * [sim(n1,n2) + sim(d1,d2) + sim(b1,b2)]`，仅对相似度超过阈值 `δ` 的技能对执行合并。
  - **训练方法**：采用“delta pass rate”引导的拒绝采样。对于每个操作，只有当候选技能使执行器在测试任务上的 pass@1 提升超过阈值 `ε` 时，才将该数据纳入监督微调。具体：
    - Create：`Δ_create = pass@1(LLM(x,s)) - pass@1(LLM(x)) ≥ ε_c`
    - Merge：`Δ_merge = 1/2 * Σ[pass@1(LLM(x_i,s)) - pass@1(LLM(x_i,s_i))] ≥ ε_m`
    - Improve：`Δ_improve = 1/2 * Σ[pass@1(LLM(x_i,s)) - pass@1(LLM(x_i,s_o))] ≥ ε_i`
  - **三种部署模式**：
    - **Offline**：离线构建通用技能库（先创建再迭代合并），推理时检索并自选择最相关技能。
    - **Online**：在线模式下逐步创建并改进技能，无需预先数据。
    - **Hybrid**：结合离线库与在线细化，先检索再迭代改进。

### 6. 论文的主要结论与发现
- SkillComposer 在所有 benchmark 上一致优于基线：SkillComposer-4B 在 τ²-Bench 上最高提升 +6.2（混合模式 85.7），在 LiveCodeBench 上最高提升 +2.5（在线模式 59.1）；且通过 4B 模型训练的技能可有效提升 27B 执行器（最高 +4.5 智能体、+3.4 代码）。
- **Skill Merge 驱动泛化**，是离线库构建中最有效的操作（+3.7 on τ²-Bench）；**Skill Improve 驱动规范**，更适合在线和混合模式。
- 技能组合能力是跨任务类型可转移的**元能力**：仅用 agent 数据训练仍能提升代码任务性能（+1.5）。
- 迭代技能演化比同等预算的独立重复采样带来更大增益（在 4B 上 k=5 时差距达 4.0 点）。
- 经过 rejection sampling 训练的模型在 OOD benchmark AppWorld 上同时提升准确率和通过率（前者 +2.5，后者 +0.2），而未训练 composer 降低通过率。

### 7. 优点
- **方法创新**：将技能构建分解为三个显式可学习操作，首次从系统角度处理规范与泛化的权衡。
- **训练高效**：通过 delta pass rate 引导的拒绝采样获得高质量 SFT 数据，无需人工标注技能。
- **部署灵活**：支持离线、在线、混合三种模式，适应不同应用场景。
- **强泛化性**：跨模型规模（4B→27B）、跨领域（Retail→Airline/Telecom）、跨任务类型（agent→code）均表现良好。
- **分析深入**：通过消融、可视化、迁移实验等手段清晰揭示了各操作的作用机制。

### 8. 不足与局限
- **计算开销**：拒绝采样需要多次推理计算 pass@k，训练数据收集成本较高，尤其在大规模场景下可能受限。
- **模型家族有限**：所有实验仅基于 Qwen3.5 系列（4B 和 27B），未在 Llama、Mistral 等其他主流模型上验证，通用性存疑。
- **技能质量风险**：虽然训练提升了技能质量，但未讨论误生成不良技能（如有毒指令、冲突指南）的风险及其缓解措施。
- **离线库构建成本**：离线库构建需要预运行全部训练任务并迭代合并，对于持续流式数据适应性有限。
- **缺乏安全性分析**：论文聚焦性能提升，未涉及技能可能引入的安全、偏见或公平性问题。
