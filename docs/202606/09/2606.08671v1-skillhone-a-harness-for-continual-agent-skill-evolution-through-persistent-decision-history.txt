# SkillHone: A Harness for Continual Agent Skill Evolution Through Persistent Decision History 

Zhiwei Li * Yong Hu 

WeChat, Tencent Inc., China 

zhiweili.jay@foxmail.com rightyonghu@tencent.com 

Abstract 

Agent skills extend language-model agents with task-specific procedures, scripts, and ref-erences, but the tasks and environments they target continually change. Existing methods improve skills in bounded runs and retain only the final artifact, discarding the decision his-tory that later agents need to interpret prior revisions, evaluations, and rejected alternatives. We introduce SkillHone, a harness for contin-ual agent skill evolution grounded in persis-tent decision history. SkillHone pairs skill re-visions with evaluation-side evidence that sup-plies practice feedback, recording structured histories of diagnoses, revisions, evidence, and outcomes. Role-separated subagents run candi-date skills on practice probes with redacted re-porting and propose revisions informed by prior decisions, enabling cross-session refinement without rediscovering past rationale. We evalu-ate SkillHone on deep-research benchmarks in a raw open-web setting, where agents are not given an integrated search stack and must orga-nize retrieval through portable skills. We com-pare against a deep-research agent backed by commercial retrieval services. With Qwen3.6-35B-A3B as the evaluation-time backbone, the resulting skills outperform the deep-research agent by 15.8 points on GAIA and 3.2 points on WebWalkerQA-EN, while also exceeding prior skill-evolution methods. 

1 Introduction 

Skills have emerged as a central mechanism for extending LLM-based agents. A skill is a named, loadable bundle of task-specific procedures, scripts, references, and output conventions that an agent can invoke for a particular class of work (Xu and Yan, 2026; Ling et al., 2026). In deployed agent systems such as Claude Code, Codex, and Her-mes, skills serve as a primary unit of specialization.

However, a useful skill is not a static artifact. APIs change, task distributions shift, and new failure modes appear, so both the skill and the evidence used to diagnose its failures must remain main-tainable. The central methodological question is therefore how later agents can continue improving a skill across many development sessions without losing the rationale behind earlier changes. iterative mutation, evaluation, and selection over skills, prompts, tool descriptions, and code using GEPA (Agrawal et al., 2026). These routes are useful for producing or improving a skill, but they largely treat improvement as a bounded run whose main output is a better artifact. This artifact-centered view loses the decision context needed for continual maintenance. A later agent may inherit the latest skill, but not the struc-tured history explaining which diagnosis motivated a revision, which alternatives were rejected, what evaluation evidence was observed, and why an out-come was accepted. The same problem applies to the evaluation evidence that supports skill improve-ment: when later agents rely on probes, validators, traces, or redacted reports, they also need to know which failure mode the evidence reflects and how it shaped prior decisions. As illustrated in Figure 1, optimization based only on the latest skill snapshot may repeat old fixes, undo useful changes, or keep optimizing against feedback whose rationale has been lost. We introduce SkillHone, an agent-facing harness for continual skill evolution in skill-enabled agent frameworks. SkillHone maintains the evolving skill together with evaluation assets that support its im-provement, and treats skill revisions and evaluation evidence as decision-bearing. For skill revisions, the history records the diagnosis, proposed revision, evaluation evidence, and outcome. For evaluation evidence, it records which probes or reports were used, which failure mode they exposed, and how the evidence justified the outcome. Thus, later agents inherit not only the current skill and current evaluation assets, but also the decisions that shaped both. This history is accumulated from many local de-cisions made during diagnosis, revision, and evalu-ation. SkillHone therefore uses role-bounded sub-agent dispatches, rather than a fixed multi-agent system, to generate these records as the loop runs: it only assumes an agent runtime that can launch scoped dispatches with separated permissions. This makes the harness portable across skill-enabled runtimes such as Claude Code, Codex, and Her-mes, while keeping optimization separated from unredacted evaluation assets. Evaluation subagents can inspect probes and traces and run candidate skills, but return only redacted evidence to the op-timization side. Optimization subagents use this evidence and prior decision records to form diag-noses and draft skill revisions, without accessing hidden probe targets or validators. Across sessions, the linked history of diagnoses, revisions, evidence, and outcomes allows later agents to continue skill evolution without re-deriving prior decisions. We evaluate SkillHone on deep-research bench-marks in the raw open-web setting, where agents must organize retrieval without a pre-integrated search stack. Against skill-evolution baselines and a deep-research agent with commercial retrieval, SkillHone obtains the strongest average results on GAIA and WebWalkerQA-EN under a fixed evaluation-time backbone.

2 Method 

2.1 Problem Setup 

In this paper, a skill is a loadable artifact that pack-ages the instructions, procedures, references, and output conventions an agent uses for a class of tasks. SkillHone keeps the model backend and external tools fixed; its goal is to help agents maintain and improve this skill artifact across repeated develop-ment sessions. SkillHone maintains two linked repositories. The skill repository stores the skill bundle be-ing revised. The skill-evaluation repository stores practice probes, oracle targets, validators, traces, redacted reports, and other evaluation assets that provide feedback for skill improvement. The opti-mization side observes only redacted reports, not unredacted targets, validators, or traces. 2.2 SkillHone Harness 

As shown in Figure 2, SkillHone runs on an agent runtime that dispatches fresh subagents into two permission-bounded sides. These are dispatch roles rather than fixed staffs of named workers: the run-time assigns each dispatch to an optimization or evaluation role, routes evidence between the two sides, and writes the resulting records into the per-sistent decision history. Appendix D (Table 2) lists the recurring dispatch patterns and their permission boundaries. We formalize the harness as 

M = ( Topt , Teval , D),

where Topt is the Optimization Subagent Team, 

Teval is the Evaluation Subagent Team, and D is the runtime dispatcher. The optimization side maintains the skill reposi-tory, which contains the current SKILL.md , scripts, references, templates, and other skill assets. Op-timization dispatches can inspect the current skill, read prior decision records and failure summaries, form diagnoses, propose typed revisions, review candidate changes, and record accepted outcomes. Typical dispatches include diagnosing failures from redacted reports, integrating prior resources, proposing scoped revisions, and drafting patches to St.The evaluation side operates on the skill-evaluation repository, which contains probes, vali-dation suites, regression sets, traces, and evaluation reports. Evaluation dispatches can inspect oracle targets and traces, run the current skill on probe items, analyze failures, and produce redacted re-ports. They cannot write to the skill repository. Conversely, the optimization side can change the skill but cannot access unredacted probe targets, validators, or execution traces. This separation pre-vents practice feedback from becoming a direct memorization target. The runtime-level dispatcher is a message router rather than a decision authority. At each step, it ob-serves the public state of the skill, the failure sum-maries, and the two repositories; dispatches role-specific subagents; routes redacted evidence and review requests; and records accepted outcomes. Acceptance decisions are made using the evidence available to the appropriate side. This keeps opti-3mization and evaluation separated while allowing both the skill and the practice feedback to evolve. 

2.3 Persistent Decision History 

A central design goal of SkillHone is to make skill evolution persistent across sessions. A later agent should not receive only the latest skill snapshot. It should also inherit the structured record of why the skill changed, which alternatives were consid-ered, how candidates were evaluated, and which outcomes were accepted or rejected. SkillHone represents each development step as a decision record 

ht = ( qt, r t, e t, o t),

where qt is the diagnosis, rt is the candidate revi-sion, et is the redacted evaluation evidence, and 

ot is the outcome. The outcome may accept the revision, reject it, request further changes, or defer the diagnosis. The decision history up to iteration 

t is 

Ht = {h1, . . . , h t}.

The decision record is more than a version diff. A diff shows how files changed; a decision record links that change to the problem it targeted, the evidence used to evaluate it, and the final decision. This distinction is important for continual evolution. When the environment changes, later subagents can inspect Ht to determine whether a failure is new, whether a similar fix was already attempted, and why a previous alternative was rejected. Given a new redacted report ˜Et, optimization subagents retrieve relevant prior records from H<t .They may revisit an old diagnosis, revise a rejected patch under new evidence, or avoid a redundant edit that has already failed. Thus the decision history acts as a persistent memory for the evolution pro-cess itself. It enables agents to audit prior changes and continue skill improvement without re-deriving the same diagnoses.

3.1 Benchmarks and Evaluation Settings 

We first instantiate this comparison in deep research because information seeking is one of the most broadly used agent workloads and naturally bene-fits from reusable skills. Unlike closed skill bench-marks that test isolated tool routines, deep-research tasks require agents to combine search, navigation, extraction, verification, and recovery across chang-ing web sources. This makes the setting a useful stress test for continual skill evolution: external resources are rate-limited, fail intermittently, and change format, so agents must diagnose failures, revise skills, and preserve the evaluation evidence behind those revisions rather than rely on a one-shot artifact. We use two open-domain benchmarks: GAIA (Mialon et al., 2024) and the English split of WebWalkerQA (Wu et al., 2025). GAIA empha-sizes multi-step information seeking and factual grounding across heterogeneous sources, while WebWalkerQA-EN focuses on web navigation and information extraction. We compare two evaluation settings: • Curated search : a deep-research agent uses commercial retrieval services and does not maintain its own search stack. • Raw open-web : agents receive no pre-integrated search tools. 3.2 Baselines 

The raw open-web comparison starts from a shared existing-skill pool. We construct this pool from search-related community skills selected by an agent from ClawHub and SkillHub; Appendix A lists the selected skills. The pool is provided to all comparable raw open-web systems for a fair comparison; the systems differ in whether and how they adapt these starting skills. The raw open-web comparison contains one direct-use setting and three skill-development sys-tems: • Existing-Skills : directly loads the skills as a portable skill bundle, without further opti-mization. • Skill-Creator : applies built-in iterative skill optimization. • Hermes-SE : applies reflective skill optimiza-tion. • SkillHone : runs the full harness with role-separated subagents, redacted evaluation feed-back, and persistent decision history. All raw open-web systems produce the same portable skill-bundle interface: a SKILL.md entry file with optional scripts, references, templates, and supporting resources. They differ only in how the bundle is developed. We separate the controller that optimizes skills from the backbone that executes and evaluates them. Claude Opus 4.6 3 serves as the development-time controller that reads feedback, forms diag-noses, and revises skill bundles. Qwen3.6-35B-A3B (Qwen Team, 2026) serves as the execution and evaluation backbone for running skills during development and for producing the final benchmark scores. 

3.3 Main Results 

Table 1 reports the final benchmark results. Skill-Hone achieves the best average score on both benchmarks. On GAIA, it reaches 64 .6% , outper-forming the deep-research agent by 15 .8 points without pre-integrated search tools at evaluation time. On WebWalkerQA-EN, it reaches 66 .4% , im-proving over the same agent by 3.2 points. The gains are most visible on harder splits, where the agent must decide how to search, extract ev-idence, verify intermediate findings, and recover from failed attempts. This suggests that the evolved skill improves the agent’s problem-solving proce-dure, not merely its access to search routines. 4 Related Work 

Agent skills and skill acquisition. Agent skills have recently emerged as a practical abstrac-tion for extending LLM-based agents with task-specific procedures, scripts, references, and conven-tions (Xu and Yan, 2026; Ling et al., 2026). Recent benchmarks further study when skills help agents across general tasks and software-engineering set-tings (Li et al., 2026; Han et al., 2026). A re-lated line of work focuses on acquiring new skills, including automated discovery from exploration and feedback (Yang et al., 2025), reinforcement-learning-based evolution (Vishe et al., 2026; Xia et al., 2026), and domain-specific skill construc-tion (Liu et al., 2026). These studies primarily ask how agents can obtain useful skills or measure whether skills improve task performance. Skill-Hone addresses a complementary question: once a skill exists and becomes part of an agent’s work-ing environment, how should later agents continue maintaining it as tasks, tools, APIs, and failure modes change? Our focus is therefore not only on producing an improved skill, but also on pre-serving the optimization context that makes future improvement possible. 

Optimization of prompts, systems, and skills. 

A second line of work optimizes language-system components through reflection, search, or program-matic compilation. Prompt optimization methods use LLMs to propose and select improved instruc-tions (Zhou et al., 2022; Yang et al., 2024a), while Reflexion-style agents use verbal feedback to im-prove behavior across attempts (Shinn et al., 2023). DSPy frames LM pipelines as declarative programs that can be compiled and optimized (Khattab et al., 2023). More closely related, GEPA performs reflec-tive evolution over components of a language sys-tem by generating variants, evaluating them, and se-lecting improved candidates (Agrawal et al., 2026). Hermes-SE applies this style of optimization to skills, prompts, tool descriptions, and code. These methods provide strong mechanisms for improving artifacts within an optimization run. SkillHone is designed for a different axis of improvement: it treats skill evolution as an ongoing maintenance process in which later agents inherit not only the lat-est artifact, but also a structured history of failures, candidate revisions, evaluations, and outcomes. 

Multi-agent collaboration and agent develop-ment workflows. Multi-agent LLM systems show that role specialization can improve com-plex collaborative problem solving (Talebirad and Nadiri, 2023; Guo et al., 2024; Wang et al., 2024; Tran et al., 2025). Frameworks such as MetaGPT organize agents into specialized roles for produc-ing software-like artifacts (Hong et al., 2024), and recent work studies LLM agents in software-engineering environments where tasks are naturally expressed through problem diagnosis, code mod-ification, and testing (Jimenez et al., 2024; Yang et al., 2024b; Xia et al., 2025). SkillHone shares the broad principle of role separation, but uses it for continual skill maintenance rather than one-shot artifact production. Its subagents are created on demand by a lightweight dispatcher and are separated by permission: optimization subagents propose skill revisions, while evaluation subagents test candidates and return redacted evidence. This separation allows the harness to accumulate agent-readable decision history while reducing leakage from evaluation assets to the optimization side. 

5 Conclusion 

We introduced SkillHone, an agent-facing harness for continual agent skill evolution. SkillHone main-tains skills as evolving bundles rather than one-shot artifacts. It dispatches role-specific optimization and evaluation subagent teams, uses redacted prac-tice feedback, and records persistent decision his-tories that link problems, revisions, evidence, and outcomes. Later agents can reuse this history to audit prior changes, avoid redundant edits, and con-tinue development across sessions. In deep research under the raw open-web setting, SkillHone develops skills without pre-integrated search tools that outperform iterative and reflective skill-development baselines. It also exceeds the deep-research agent on GAIA and WebWalkerQA-EN under a fixed evaluation-time agent back-bone. These results suggest that persistent decision history helps agents turn practice feedback into stronger skill bundles. 

Limitations 

Our experiments are conducted on English-language benchmarks. Extending evaluation to other languages remains future work. Additionally, 6SkillHone currently evolves a single skill in isola-tion; jointly evolving multiple interdependent skills is not addressed in this work.

B Evaluation Repository and Redaction 

SkillHone uses a skill-evaluation repository for practice feedback during skill development. The repository stores probe task instances, oracle tar-gets, validators, provenance metadata, and execu-tion traces. Probe items may come from existing sources such as human-written cases, historical failures, or environment-derived checks. It only assumes that evaluation subagents can run the cur-rent skill against available probe items and export redacted reports to the optimization side. For each probe item, we denote the task instance by xk, the oracle target by yk, the validator by ϕk,and the provenance metadata by πk. Running the current skill produces 

mk = ϕk(A(xk; St), y k),

where mk is the validation outcome. The optimiza-tion side receives only a redacted problem report 

˜Et = Redact( {xk, y k, ϕ k, π k, m k}Kk=1 ),

which summarizes failure modes, aggregate out-comes, and diagnostic hypotheses without expos-ing unredacted targets, validators, or traces. Fi-nal benchmark evaluation remains outside the opti-mization loop.
