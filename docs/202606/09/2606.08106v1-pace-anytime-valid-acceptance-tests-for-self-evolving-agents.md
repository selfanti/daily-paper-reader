## 摘要
自我进化智能体通过反复向其自身的提示、技能或工作流程提出更改，并保留那些在小型保留集上得分更高的更改来进行改进。几乎所有工作都集中在生成候选的提议者上；我们认为薄弱环节是接受者，即决定是否提交更改的规则。针对同一个有噪声的开发估计进行数百次应用，无处不在的“如果分数上升则保留”规则是一种失控的自适应多重测试：智能体实际上在自我进行p-hacking，累积了错误的提交，导致其搅动和漂移，而非改进。我们重新将提交视为序贯假设检验，并提出了PACE（成对任意有效提交评估），一种无需训练的、任意有效的提交门控。每个候选与现有模型在相同实例上进行对比，仅当通过“测试下注”e过程积累决定性证据时才提交，提前停止以节省评估次数，并在可选停止的情况下将每个候选的错误提交概率控制在用户设定的水平（每决策保证）。在Qwen2.5智能体（0.5B-3B）于GSM8K、SVAMP和ARC-Challenge上在提示层面进行自我进化的实验中，当真实改进隐藏在噪声提议中时，贪婪接受提交了30-42%错误和10-33%有害编辑，而PACE提交了真实改进且几乎不提交其他内容，匹配了贪婪方法的保留集准确性，但方差显著降低，评估成本降低约18%。在没有真实改进可用时，贪婪方法每次运行提交了13-21个虚假自我修改（72-100%错误），使最脆弱的智能体性能下降4.9个点，而PACE保持基线。自我进化的可靠性取决于接受者，而不仅仅是提议者。

## Abstract
Self-evolving agents improve by repeatedly proposing changes to their own prompts, skills, or workflows and keeping those that score higher on a small held-out set. Almost all effort has gone into the proposer that generates candidates; we argue the weak point is the acceptor, the rule that decides whether to commit a change. Applied hundreds of times against the same noisy dev estimate, the ubiquitous "keep it if the score went up" rule is uncontrolled adaptive multiple testing: the agent effectively p-hacks itself, accumulating false commits that make it churn and drift rather than improve.   We recast committing as a sequential hypothesis test and propose PACE (Paired Anytime-valid Commit Evaluation), a training-free, anytime-valid commit gate. Each candidate is compared to the incumbent on identical instances and committed only when a testing-by-betting e-process accumulates decisive evidence, stopping early to save evaluations and controlling each candidate's false-commit probability at a user-set level even under optional stopping (a per-decision guarantee).   On Qwen2.5 agents (0.5B-3B) self-evolving at the prompt level on GSM8K, SVAMP, and ARC-Challenge, greedy acceptance commits 30-42% false and 10-33% harmful edits when a genuine improvement is hidden among noisy proposals, while PACE commits the real one and essentially nothing else, matching greedy's held-out accuracy at sharply lower variance and about 18% lower evaluation cost. With no real gain available, greedy commits 13-21 spurious self-modifications per run (72-100% false) and degrades the most fragile agent by 4.9 points, while PACE holds at baseline. Reliability of self-evolution depends on the acceptor, not only on the proposer.

# PACE: Anytime-Valid Acceptance Tests for Self-Evolving Agents 论文总结

## 1. 核心问题与整体含义

- **研究动机**：自我进化智能体通过反复修改自身提示、技能或工作流程来改进，但几乎所有研究都聚焦于如何生成更好的候选修改（提议器），而忽略了决定是否接受修改的环节（接受器）。当前普遍采用的接受规则——“如果开发集分数上升则保留”——本质上是无控制的自适应多重假设检验：在数百次重复使用同一个有噪声的小型开发集进行决策时，智能体相当于在进行 *p-hacking*，累积大量虚假提交（false commits），导致修改只会带来性能搅动和漂移，而非真正改进。
- **整体含义**：该工作诊断了自我进化循环中被忽视的弱点，并提出了统计严格的解决方案。不仅要关注提议器，更应重视接受器，可靠性取决于两者。

## 2. 方法论

- **核心思想**：将提交决策重铸为序贯假设检验，使用配对评估和测试下注（testing-by-betting）的 e-过程（e-process），实现任意有效（anytime-valid）的门控。每个候选与当前配置在相同实例上成对比较，仅当证据累积超过阈值时才提交，控制每候选错误提交概率。
- **关键技术细节**：
  - **配对评估**：只考虑不一致对（一个正确、另一个错误），消除示例难度差异的影响。在零假设“候选不优于当前”下，不一致对中候选正确的概率为 1/2。
  - **e-过程更新**：初始财富 E0 = 1，每次遇到不一致对，按公式更新：E ← E·(1 + λ(2wi − 1))，其中 wi = 1 表示候选正确且当前错误，wi = 0 表示相反。λ 为赌注比例（实验中 λ = 0.5）。
  - **提交条件**：当 E ≥ 1/α 时（α 为用户设定的错误提交概率，如 0.05）提交修改。在零假设下，E 是均值为 1 的非负鞅，由 Ville 不等式保证即使任意停止（optional stopping）也能控制假阳性概率 ≤ α。
- **算法流程**（文字说明）：
  1. 当前配置 c，开发集 D。
  2. 对每一轮 t，提议器生成候选 c‘。
  3. 初始化 E = 1，从 D 中抽取实例，成对评估 c 和 c’。
  4. 若两者结果相同（平局）则跳过；若不一致，则根据 wi 更新 E。
  5. 当 E ≥ 1/α 时提交 c‘，否则若实例耗尽则拒绝。
- **特点**：无需训练，仅消耗配对结果，不依赖提议器内部机制。

## 3. 实验设计

- **数据集与场景**：
  - GSM8K（数学推理）、SVAMP（算术推理）、ARC-Challenge（多选科学）。
  - **控制场景**：在初始提示中植入一个故意有害的指令，使得提议器能通过移除它获得真实的大幅改进，用于测试接受器能否区分真实增益与噪声。
  - **随机场景**：agent 采样温度设为 0.7（典型部署场景），无真实改进可用，测试接受器是否避免追逐采样噪声。
- **Benchmark与对比方法**：
  - **贪婪接受** (Greedy)：开发集得分上升即保留。
  - **固定 n 配对检验** (Paired fixed-n)：设定固定配对数量后做 McNemar 检验。
  - **在线 FDR 控制** (Online-FDR)：使用衰减记忆的在线错误发现率控制。
  - **PACE**：本文提出的任意有效门控。
- **模型与评估**：Qwen2.5-Instruct（0.5B、1.5B、3B）。使用独立审计池（n=120，温度 0）测量真实性能，开发集 n=40 用于决策。所有结果平均 5 种子（控制场景）或 3 种子（随机场景）。

## 6. 主要结论与发现

- **贪婪接受导致大量虚假和有害提交**：控制场景下 30–42% 虚假，随机场景下 72–100% 虚假，小模型（0.5B）性能下降 4.9 点。
- **PACE 几乎完全消除虚假提交**：在所有场景中虚假提交率约 0%（审计标签，5/5 种子 0 个虚假），同时保持真实改进。
- **PACE 方差更低、评估成本更低**：在 3B 模型上，PACE 的 Δ = +0.74 ±0.04，而贪婪为 +0.54 ±0.30；评估问题数减少约 18%。
- **无真实改进时，PACE 保持基线性能，而贪婪持续搅动**。
- **核心结论**：自我进化的可靠性不仅取决于提议器，也取决于接受器，PACE 提供了统计安全的默认提交决策。

## 7. 优点

- **方法简洁有效**：约 10 行代码，无需训练，即插即用，适用于任何可对同一实例评分的自我修改循环。
- **统计理论保证**：任意有效控制每候选错误提交概率，允许可选提前停止而不破坏控制。
- **实验设计隔离了接受器的影响**：选择提示级进化作为最小测试用例，清晰归因。
- **跨任务、跨模型尺度、跨场景广泛验证**，结果一致。
- **关注实际部署问题**：虚假提交导致的不稳定性和计算浪费是实际风险，PACE 提供了实际可行的缓解方案。

## 8. 不足与局限

- **仅验证了提示级进化**，更高层的进化（技能、代码、多智能体工作流）尚未测试，尽管接口通用。
- **仅使用单一模型系列（Qwen2.5）和单一提议器**，提议器无关性仅从接口论证，未实证。
- **审计标签为有限样本估计**：尽管观察到的差距极大，但并非完全精确的地面真实。
- **每候选控制而非运行级家族错误控制**：虽然足够消除贪婪的病理，但多个错误提交的风险随候选数线性增长（但 α 很小，实践中可信）。
- **对于非常微小的真实增益会牺牲召回率**（例如保留 78% 而非 100%），不适合“错过一个微小改进比误提交一个修改代价更高”的场景。
- **依赖开发集的覆盖范围**，无法保证改进在开发集外泛化。
