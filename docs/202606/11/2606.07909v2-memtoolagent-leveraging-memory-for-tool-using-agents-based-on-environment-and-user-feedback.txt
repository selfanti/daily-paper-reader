Abstract 

Modern large language model (LLM) agents can use external tools to help users solve com-plex tasks. However, for problems that require learning from long-term historical events or from previous agent-environment interactions, LLM agents are required to use memory mech-anisms to store and retrieve experiences. While sophisticated memory systems exist for dia-logue agents, few studies have empirically ex-amined how to improve agents’ tool-using ca-pabilities through past user-agent conversations. We propose MemToolAgent, a framework that improves tool use through memory manage-ment. Our approach contains a memory ex-traction module that processes past experiences into structured memory entries, and a retrieval module that dynamically selects a subset of the stored memory entries. This enables more personalized and accurate responses aligned with user preferences and feedback without re-quiring LLM fine-tuning. In summary, this work has three main contributions: (1) a uni-fied memory entry format that improves both general-purpose and personalized tool use with-out LLM fine-tuning, (2) a reflection-based memory extraction that uses environment and user feedback to distill wrong executions into critiques to store, and (3) a retrieval module that chooses how many past experiences to use based on the memory similarity distribu-tion. MemToolAgent achieves 29%, 80%, and 17% relative improvements compared to strong baselines on the WorkBench, NESTFUL, and PEToolBench benchmarks, respectively. 

1 Introduction 

Combining powerful LLMs with additional compo-nents like memory systems, workflow mechanisms, APIs, and other tools to create LLM agents is an emerging trend. These agents are designed to solve complex tasks that require sequential reasoning

and multi-step planning. In this context, tools are functions utilized by agents to interact with their environment. Integrating LLMs with external tools and APIs allows these language models to leverage various systems, enabling end-users to interact with those systems through a chat interface. An LLM agent’s tool use ability, i.e., using appropriate tools for solving a task, is an important capability. Despite significant advancements in recent years, fixed context windows of LLMs limit maintaining coherence and thematic continuity during long real-world conversations (Tan et al., 2025). Due to long sequences containing abundant irrelevant informa-tion, the attention mechanism loses its effective-ness, which is not desired for critical scenarios in education and healthcare. As a result, integrat-ing memory mechanisms into agents is crucial for long-term interaction ability. LLM agents need reliable external feedback to correct themselves (Kamoi et al., 2024), and memory can provide this feedback by storing and retrieving past interactions. Using experiences allows agents to more accurately anticipate user needs and learn from mistakes. Integrating memory into an LLM agent supports two main types of tasks: personalized and general-purpose tool use, as shown in Figure 1. Users utilize the same tools in particular ways, such as using different input parameters, or they might use different tools to achieve the same goal. In scenar-ios where the agent has access to tools with similar functionalities, it requires user context to correctly execute the task. Storing the past interactions in the memory allows for better personalization. For instance, personalization can resolve the tool-level ambiguity if "create a ticket" could refer to Zendesk ticket creation or Jira issue creation. Regarding general-purpose tool use, there are common error patterns in tool use agents, such as dependency vio-lation, parameter hallucination, and premature tool calling. Having past experiences in memory can help the agent use the correct parameters for tools

Our work offers three main contributions: (1) a unified memory entry format of storing past experi-ences for better general-purpose and personalized tool use performance without LLM fine-tuning, (2) an environment and user feedback-based reflec-tion approach for memory writing to distill failed past experiences into critiques, and (3) a retrieval module based on similarity distribution that dynam-ically selects relevant memory entries to guide the agent for the current task. 

2 Related Work 

There has been a growing interest in memory man-agement for LLM agents in recent years. A re-lated earlier idea is Case-based reasoning (CBR) (Aamodt and Plaza, 1994) where systems retrieve and use similar past cases (experiences) to solve new problems. MemGPT (MemoryGPT) (Packer et al., 2023) introduces an OS-inspired memory hierarchy with two memory types called the main context and the external context. More recent work includes graph-based architectures to organize the information more flexibly and expressively. In Mem0 (Chhikara et al., 2025), the memory is mod-eled as a directed knowledge graph to express com-plex relationships. This allows dynamic and more advanced storage and retrieval. A-MEM (Agen-tic Memory) (Xu et al., 2025b) proposes a mem-ory based on the Zettelkasten knowledge manage-ment system. The memory entries are linked as interconnected networks, and the existing entries are updated based on new experiences. Similarly, Zep (Rasmussen et al., 2025) and Cognitive Weave (Vishwakarma et al., 2025) are based on temporally aware knowledge graphs and Spatio-Temporal Res-onance Graphs, respectively. Reflexion agents (Shinn et al., 2023) turn task feedback into self-reflections that are stored in an episodic memory for subsequent runs. Reflective Memory Management (RMM) (Tan et al., 2025) uses a Reinforcement Learning based approach to rank memory entries based on their utilities in past executions. PersonaAgent (Zhang et al., 2025a) combines episodic and semantic memory for an LLM agent with personalized memory that enables the agent to perform actions customized to the user. Agent Workflow Memory (Wang et al., 2024b) stores workflows in the memory to guide the agent in web navigation tasks. EXPEREPAIR (Mu et al., 2025) introduces an agent with a dual-memory mechanism for code repair. T1 (Chakraborty et al., 2025) uses a simple cache for agents to store tool call results. However, the main goal is to reuse the previously accessed data. 

M em p (Fang et al., 2025) is a procedural mem-ory framework that extracts both fine-grained in-structions and high-level abstractions from past agent trajectories. Memento (Zhou et al., 2025) stores past experiences in an episodic memory, learns and continually updates a case-selection pol-icy based on environmental feedback. Training-Free GRPO (Cai et al., 2025) is a training-free alternative to agentic RL that optimizes an experi-ence library consisting of natural-language experi-ential knowledge. TOOLMEM (Xiao et al., 2025) stores memories of tool capabilities from previous interactions and helps the agent select the most suit-able tool for a given task at inference. LEGOMem (Han et al., 2025) is a procedural memory approach for multi-agent systems that distills only success-Figure 2: MemToolAgent overview with a simple restaurant booking scenario where the agent retrieves similar memories, receives feedback on an invalid time format, and generates a reflection to update its memory 

ful task trajectories into memory entries with brief reflections. ReMem agents (Wei et al., 2025) com-bine reasoning, action, and memory refinement and actively exploit, prune, and reorganize their mem-ory while solving the current task. There are in-context learning (ICL) approaches closely related to agents with memory. For exam-ple, STE (Wang et al., 2024a) creates tool use tra-jectories based on a trial-and-error approach offline and stores them in a static memory for inference. In-Context Abstraction Learning (ICAL) (Sarch et al., 2024) creates memory abstractions from sub-optimal trajectories based on human-in-the-loop learning and self-reflection. Unlike the related work, MemToolAgent ad-dresses both personalized and general-purpose tool use. It proposes a general memory entry format and stores and leverages detailed information like tool trajectories and a reflection/critique about the execution. Importantly, MemToolAgent also incor-porates user feedback as a signal to improve the quality of memory entries. 

3 Method: MemToolAgent 

MemToolAgent consists of two main components: the memory retrieval and memory extraction mod-ules. Figure 2 illustrates the general overview of our proposed approach. Memory entries, i.e., past experiences, are retrieved from the memory based on the semantic similarity between the query and the entries. Dynamic "n" Top-n Embedding-based Retrieval is proposed for retrieval. Instead of re-trieving a fixed number of memory entries for each query, the number of entries to retrieve is dynami-cally determined based on the cosine similarities be-tween the embeddings of memory entries’ queries and the test query. Then, the entries are used as in-context learning to guide the agent for the current task. To add a new entry to the memory based on the current query, an LLM-based "Memory Extraction" module provides a reflection/critique of the execu-tion, which is then stored in the memory with the query and user feedback. MemToolAgent specifies what to store in mem-ory with a general memory entry format. Each memory entry m is represented as a tuple m =(q, a, f, r ) where: • q: the user query • a = {a1, a 2, . . . , a n}: the sequence of tool calls executed by the agent • f ∈ { 0, 1}: binary user feedback indicating task success ( f = 1 ) or failure ( f = 0 )• r: reflection or critique generated by the mem-ory extractor when f = 0 (otherwise r = ∅)With binary user feedback, we simulate the sce-narios where the user does not know anything about the underlying tools/APIs and gives feedback based on the outcome. However, this feedback could eas-ily be replaced by any other type of feedback, such as text or score. Figure 3 illustrates an example memory entry with reflection. 

3.1 Memory Retrieval: Dynamic "n" Top-n Embedding-based Retrieval 

Retrieving a fixed number of memory entries for ev-ery query is not optimal since some queries might not have relevant memory entries, while there might be many similar entries for other queries. Prior work (Taguchi et al., 2025; Xu et al., 2025c) in retrieval-augmented generation (RAG) addresses the trade-off between missing information with too small k and adding redundancy with too large k, where k is the number of retrieved documents. These methods dynamically decide the number k and show that the optimal amount of retrieved con-text varies across queries. Adaptive-k (Taguchi et al., 2025) selects "k" as the largest adjacent similarity-score gap. Our Dynamic "n" approach has four main steps illustrated by the Algorithm 1. The first two steps are calculating the cosine similarity values between the query and memory entries and sorting them in descending order. The third step calculates a first-order derivative estimate of sorted similarities based on a sliding window approach where x is an array of sorted similarities, and R is considered as the "radius" of the derivative estimate. The fi-nal step is finding the peaks, i.e., sudden drops in similarity, ideally showing a class boundary. The peak corresponds to our “n” value for Dynamic "n" retrieval. The entire process is inspired by the concept of finding “knees” for designing systems (Satopaa et al., 2011). The find_peaks * function from the SciPy library is utilized for peak finding. Figure 4 illustrates an example. The blue line shows the sorted similarity values, while the red line shows the first derivative estimation. As ob-served, the locations where there are local maxima, i.e., a peak, correspond to locations with a sudden drop in similarity. Dynamic "n" retrieval has three parameters, namely R, prominence, and the peak number. R is a smoothing parameter, and larger R values result in smoother derivative estimates. Prominence is a parameter of the find_peaks func-tion to filter important peaks based on comparison of neighboring values. The peak number is simply which peak to select.

We hypothesize that finding peaks based on the derivative estimate rather than raw similarity values results in a more robust approach than Adaptive-k. Dynamic "n" robustly finds points of change or cluster boundaries on noisy similarity curves, while Adaptive-k finds the biggest drop and works well only when the curve has one clean knee.

3.2 Memory Extraction Module 

The agent needs to apply experience to the current task based on the user feedback and the context. An important observation from our experiments is that agents struggle to effectively learn from retrieved negative experiences when they are presented as raw execution traces. We hypothesize this is be-cause the agent must simultaneously understand what went wrong in the past example while solving the current task. To make the agent’s work eas-ier, a "Memory Extraction" module is proposed. It reflects on the context and user feedback to under-stand what was wrong. For wrong executions, the reflection/critique is added to the memory instead of the raw context, which might be too complicated and contain irrelevant parts. Doing the reflection as a separate step helps the agent focus only on the current task and learn from its mistakes. The memory extraction module is an LLM E that takes the following as input: (1) current query q,(2) set of available tools T = {t1, t 2, . . . , t m}, (3) agent’s response (actions) a = {a1, a 2, . . . , a n}

including tool calls, (4) environment feedback e

including tool responses, (5) user feedback f ∈{0, 1}, and (6) retrieved semantically similar mem-ory entries M′ ⊂ M . Using retrieved memory entries that are semanti-cally similar to the current user query is important. The main use case for general-purpose tool use is having different queries with common mistakes, which are only sometimes detected by the memory extractor. A similar entry with a correct reflection could help the memory extractor produce a cor-rect critique. For personalized tool use, memory extractor is slightly different. Semantically simi-lar memory entries are used to find user patterns and preferences. The memory extraction module is used to extract high-level user preferences, such as “User prefers tools under AP I _X". 

4 Evaluation 

Memory can be applied to multiple problem set-tings in tool-using agents. Our experiments focus on two main tasks: (1) general-purpose tool use and (2) personalization. Three benchmarks, Work-Bench (Styles et al., 2024), NESTFUL (Basu et al., 2024), and PEToolBench (Xu et al., 2025a), were used for evaluation. WorkBench and NESTFUL are general-purpose tool use benchmarks, while PEToolBench is a personalized tool orchestration benchmark. 

4.1 Benchmarks 

WorkBench is an outcome-centric tool use bench-mark that evaluates agents on workplace tasks such as sending emails and scheduling meetings. It in-cludes five databases and 27 tools. WorkBench tasks are from different domains like Analytics, Calendar, etc. The evaluation metric is accuracy, which is defined as the percentage of tasks where the outcome from the agent’s actions matches the expected outcome. The correct outcome means the correct changes in the database values. WorkBench queries were created based on pre-defined templates such as ’Delete an email from person X’ or ’Schedule a meeting with person Y’. There are 10 queries per template. Since the bench-mark has no train-test split, one query from each template is used to populate the memory. This ensures that there is at least one directly relevant memory entry for each test query. This setup is used to show the advantages of using the memory. The test set consists of 108 Analytics, 99 Calendar, 72 CRM, 81 Email, and 189 Multi-domain tasks. NESTFUL benchmark evaluates LLMs on nested sequences of tool calls, where the output of one tool call is used as input for the next tool call. It includes 1891 test queries based on MathQA (Amini et al., 2019) and StarCoder2-Instruct (Wei et al., 2024) benchmarks. The first 500 tasks are used as test set while the next 500 tasks are used to populate the memory. The benchmark uses five evaluation metrics, which are F1 scores for tool and parameter generation, Partial Sequence Matching, Full Sequence Matching, and Win Rate. Partial Sequence Matching measures how many predicted APIs with their argument-value pairs match the gold API sequence, Full Sequence Matching eval-uates whether the model produces the exact full sequence, and Win Rate assesses whether the pre-dicted APIs lead to the gold answer when executed. PEToolBench is a personalized tool orchestra-tion benchmark. Each query comes with an instruc-tion history consisting of 80-100 past user queries, where each past query includes a binary rating (1 for correct/desired execution, 0 otherwise). For our experiments, this instruction history forms the memory, where each memory entry consists of a past query, its corresponding tool call, and binary user feedback. Test queries are designed such that the agent must rely on user preferences from mem-ory for correct tool execution. A subset of 591 queries is used as the test set after filtering the queries with sufficiently long instruction histories (at least 80 past queries). This makes sure that the agent needs a memory rather than using the full history in context. The benchmark uses two evalu-ation metrics: Tool Accuracy, which evaluates the model’s ability to call the appropriate tool based on user preferences, and Parameter Accuracy, which assesses whether the model generates the correct parameters.

5 Conclusion 

MemToolAgent is a memory framework that en-hances the tool-using capabilities of LLM agents by using past experiences. The results show sig-nificant improvements across multiple benchmarks in both personalized and general-purpose tool use scenarios. The main contributions of the frame-work include (1) a retrieval mechanism that dynam-ically determines the number of relevant memory entries to retrieve, (2) a memory extraction module that generates reflections on incorrect executions to help avoid similar mistakes, and (3) an overall effective approach for both general-purpose and personalized tool use based on user and environ-ment feedback. 

Limitations 

Despite the improvements in general-purpose and personalized tool use performance achieved by MemToolAgent, several limitations remain for fu-ture work. One limitation is the cold-start problem of memory. Moreover, the benchmarks used con-tain fewer than 40 tools. However, there are typ-ically more tools in real-life applications. As the number of tools increases, it is more challenging to cover the usage of all tools with memory. This could potentially make the cold-start problem more severe. Our experiments using synthetically gener-ated queries show that even a simple approach can help the cold-start problem. Another limitation emerges in conversational scenarios with multiple turns. The benchmarks used in our experiments are single-turn. For single-turn tasks, the memory retrieval is done once at the beginning of the conversation. This might not be optimal for multiple-user turn tasks where the agent needs to obtain the necessary information from the user across multiple turns. One possible solution is to extend MemToolAgent to retrieve from memory after each user turn, so that the agent can focus on the sub-task associated with that specific turn. 

Ethical Considerations 

The proposed MemToolAgent approach has been implemented and evaluated using the publicly avail-able NESTFUL, WorkBench, and PEToolBench benchmarks.
