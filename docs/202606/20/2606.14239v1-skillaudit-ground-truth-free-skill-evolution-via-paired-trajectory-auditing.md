## 摘要
智能体技能是结构化的程序包，用于指导冻结的大语言模型智能体执行专业工作流。技能在部署后很少能保持足够适用：边缘情况、API变更和部署限制只有通过使用才会显现，因此技能演化成为一种实际需求。现有方法依赖特权反馈，例如留出验证分数、隐藏测试结果或环境奖励——当从业者仅有任务描述和工作区数据时，这些信号通常不可用。我们提出了SkillAudit，一个无需真实基准反馈即可演化智能体技能的框架。其核心思想是成对轨迹审计：在每次迭代中，使用与不使用候选技能执行相同任务，从而在不借助外部标签的情况下隔离技能对智能体行为的影响。为了将行为差异转化为编辑指导，SkillAudit使用了过程对齐对比评估（PACE），这是一组评估器，将轨迹差异映射到与技能文档中特定段落相关的诊断信号。一个结构验证器，由任务规范一次性编译后固定，检查任务约束并回滚有害更新。SkillAudit通过两条流水线路由编辑：Refine从广泛有用的技能中移除噪声或无关指导，而Repair替换与任务冲突的段落。在涵盖8个专业领域的89个容器化任务中，SkillAudit实现了73.9%的平均任务奖励，优于无技能智能体（40.9%）和静态专家技能（56.7%）。这些增益是在演化过程中无需访问隐藏测试、参考解决方案或外部评分函数的情况下获得的。

## Abstract
Agent skills are structured procedural packages that guide frozen LLM agents in specialized workflows. Skills rarely remain sufficient after deployment: edge cases, API changes, and deployment constraints become visible only through use, making skill evolution a practical necessity. Existing methods depend on privileged feedback such as held-out validation scores, hidden test outcomes, or environment rewards -- signals often unavailable when a practitioner has only a task description and workspace data. We introduce SkillAudit, a framework for evolving agent skills without ground-truth feedback. The key idea is paired trajectory auditing: at each iteration, the same task is executed with and without the candidate skill, isolating how the skill changes agent behavior without external labels. To turn behavioral differences into edit guidance, SkillAudit uses Process-Aligned Contrastive Evaluation (PACE), a cluster of evaluators that maps trajectory divergences to diagnostic signals linked to specific passages in the skill document. A structural verifier, compiled once from the task specification and then fixed, checks task constraints and rolls back harmful updates. SkillAudit routes edits through two pipelines: Refine removes noisy or irrelevant guidance from broadly useful skills, while Repair replaces passages that conflict with the task. Across 89 containerized tasks spanning 8 professional domains, SkillAudit achieves 73.9% average task reward, outperforming an agent without skills (40.9%) and the static expert skill (56.7%). These gains are obtained without accessing hidden tests, reference solutions, or external scoring functions during evolution.

## 1. 核心问题与整体含义（研究动机与背景）

- **问题**：LLM智能体在执行长时序专业任务（如软件工程、科学分析）中依赖“技能”（技能包）。技能在部署后常因边缘情况、工具API变更、数据格式变化或部署约束变得不适用，因此需要技能演化。
- **现有方法的局限**：现有技能演化方法（如SkillOpt、CoEvoSkills、SkillForge、SkillClaw）均依赖外部特权信号——留出验证分数、隐藏测试结果、环境奖励、专家参考响应或历史故障日志。在真实场景中，从业者通常只有任务描述（T）和工作区数据（W），没有隐藏测试、参考解、奖励函数或人类验证脚本。
- **挑战**：如何在优化过程中完全无外部真值信号（ground-truth-free）的情况下，让智能体技能自主改进。

## 2. 方法论：核心思想、关键技术细节

- **核心思想：成对轨迹审计（Paired Trajectory Auditing）**  
  在每次迭代中，对同一任务分别执行**带技能**的版本（τw）和**不带技能**的版本（τwo），通过对比两条轨迹的行为差异，隔离技能对智能体的影响，无需外部标签。

- **关键技术组件**：
  - **PACE（过程对齐对比评估）**：由12个评估器组成的簇，从四个维度评估轨迹差异：
    - 过程遵循性（Process Adherence）
    - 工件证据（Artifact Evidence）
    - 一致性（Consistency）
    - 效果差异（Effectiveness Delta）
    每个评估器输出结构化判断（Pass/Fail/Warning）、锚定到技能文档具体段落的动作信号，以及保护提示（不可修改的段落）。
  - **锚定验证器（Anchor Verifier）**：从任务规范一次性编译得出固定脚本，检查硬性结构约束（文件存在、格式合规、可计算值等），并在演化全程锁定。提供确定性、可复现的边界检查，防止PACE评估漂移。
  - **判决机制**：
    - 任何“hurt”报告直接否决更新（veto）。
    - 无hurt时，工件证据评估器决定性；余下按多数票归为helped/inert。
    - 由收敛条件（连续两次无hurt、验证器通过、无剩余手术目标）或最多5次迭代终止。
  - **双策略演化**：
    - **Refine流水线**（减为主）：适用于技能核心正确但含噪声的情形，默认减去错误/冗余内容，保护有效段落，一旦出现hurt立即停止。
    - **Repair流水线**（换补为主）：适用于技能核心与任务冲突（API过时、误导性指导）的情形，用τwo中的已验证片段替换有害段落，允许诊断后填补知识空缺。

- **算法流程**（算法1）：  
  1. 任务解析器分析T和S0，生成结构化任务规范。  
  2. 编译锚定验证器，进行预评估路由（Refine/Repair）。  
  3. 迭代循环（最多5次）：  
     - 成对执行 → 聚合PACE和验证器结果 → 判决 → 若为helped则执行Skill Iterator编辑；若为hurt则回滚；若为inert则推迟。  
     - 检查收敛条件，提前终止。  
  4. 返回最终技能S*。

- **性能提升**：SkillAudit在89个任务上平均任务奖励73.9%，显著超过无技能（40.9%）和静态专家技能（56.7%），增益+33.0pp和+17.2pp。

- **可观测性边界**：决定演化成功与否的关键因素是技能知识的**可观测性**而非领域。可执行、可检验的知识（如API调用、数学公式）演化良好（~80%）；而领域流程类知识（domain procedure）因不留下结构痕迹，演化困难（77%失败任务携带此类标签）。
- **编辑行为模式**：成功演化技能呈现减噪、去教程、去硬编码、补充I/O合同、添加导航索引等规律性操作。技能最终收敛到信息密集的中等长度。
- **核心教益**：好的技能应该是可验证的执行合同，而非教程；应避免庞大通用或角色扮演式技能包；多技能共存时需要路由索引。

## 7. 优点

- **新颖问题设定**：首次在完全无真值反馈的条件下实现技能演化，填补了现有方法依赖特权信号的空缺，更贴近真实部署场景。
- **成对轨迹审计**：直接利用轨迹差异作为演化信号，无需外部标注，清洁有效。
- **双评估架构**：结合软性PACE评估（提供段级别精细诊断）与硬性锚定验证器（确保结构约束），取得稳定性和灵活性的平衡。
- **双策略路由**：针对不同问题类型设计Refine/Repair，避免一刀切带来的破坏或无效修补。
- **实验设计严谨**：演化与评估严格分离，保证无信息泄漏；使用容器化环境确保可复现性。
- **大量定性分析**：详细的结构分析（编辑词汇、长度变化、案例研究）提供了对技能演化机制和局限的深入洞察。

## 8. 不足与局限

- **缺少组件消融实验**：未量化评估PACE各维度、锚定验证器、双路由策略的各自贡献，限制了架构设计的可解释性。
- **路由启发式的可靠性未量化**：任务分配为Refine/Repair基于初步兼容性检查，其误判率及下游成本未报告。
- **可观测性边界客观存在**：对领域流程类知识（无结构痕迹）技能几乎无法自我修复，必须借助人类或更强Oracle。这限制了方法的适用范围。
- **收敛假设理想化**：系统在无动作时过早收敛，可能导致未充分优化的技能被接受（如过长的技能未被压缩）。最大迭代次数5次可能不足。

- **无动态环境测试**：未模拟工具API变更、数据流漂移等典型演化场景，仅在静态任务集上评估。
- **实验统计性不足**：定性观察占主导，缺乏置信区间或显著性检验。
