## 摘要
近期大语言模型智能体的进展在自动化数据科学领域取得了令人瞩目的突破。然而，现有方法仍受限于静态动作集和缺乏原则性的长程上下文管理，这阻碍了其跨任务积累可复用经验以及在多阶段、迭代的数据科学流程中可靠运行的能力。为解决这些挑战，我们提出了EvoDS——一种自演化的自主数据科学智能体，它通过学习扩展自身技能并通过智能体强化学习自适应管理长期上下文。具体而言，EvoDS引入了两项关键策略：(1) 自主技能获取机制，使智能体能够合成、验证并复用可执行技能；(2) 自适应上下文压缩策略，将上下文管理视为一个学习到的控制问题而非被动截断。这些策略在一个两阶段多智能体训练方案中协同运作，使EvoDS能够随时间自主改进。理论上，我们证明了EvoDS的层次化设计减少了工具选择错误，且其优化目标与信息瓶颈原理一致，确保了上下文的高效利用。实验上，EvoDS在四个不同基准测试上平均优于最先进的开源数据科学智能体28.9%，同时消除了令牌溢出失败。我们的代码和数据可在 https://github.com/usail-hkust/EvoDS 获取。

## Abstract
Recent progress in Large Language Model (LLM) agents has enabled promising advances in automated data science. However, existing approaches remain fundamentally limited by their static action sets and lack of principled long-horizon context management, hindering their ability to accumulate reusable experience across tasks and operate reliably in multi-stage, iterative data science pipelines. To address these challenges, we introduce EvoDS, a self-evolving autonomous data science agent that learns to expand its skills and adaptively managing long-term context through agentic reinforcement learning. Specifically, EvoDS introduces two key strategies: (1) Autonomous Skill Acquisition (ASA) mechanism, which enables agents to synthesize, validate, and reuse executable skills; and (2) Adaptive Context Compression (ACC) strategy, which treats context management as a learned control problem rather than passive truncation. These strategies are orchestrated within a two-stage multi-agent training scheme, enabling EvoDS to autonomously improve over time. Theoretically, we prove that EvoDS's hierarchical design reduces tool-selection error, and its optimization objective aligns with an information bottleneck principle, ensuring efficient context use. Empirically, EvoDS outperforms state-of-the-art open-source data science agents by an average of 28.9% across four diverse benchmarks while eliminating out-of-token failures. Our code and data are available at https://github.com/usail-hkust/EvoDS.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：现有的大语言模型（LLM）驱动的数据科学智能体大多依赖预定义的工作流和固定的动作空间（如DS-Agent、AutoKaggle、DeepAnalyze等），缺乏从经验中抽象可复用技能的能力，且无法有效管理多阶段、迭代式数据科学任务中产生的长程上下文（导致“Lost-in-the-Middle”现象和令牌溢出失败）。这限制了智能体的持续改进和长期可靠性。
- **整体含义**：为了实现真正的自主数据科学智能体，需要同时解决两个紧密耦合的挑战：(1) **自动技能获取**——从经验中合成、验证并复用可执行技能，使动作空间随时间动态扩展；(2) **长程上下文管理**——在有限上下文窗口内选择性保留任务关键信息，抑制无关细节。EvoDS正是为此而设计，通过层次化多智能体架构、自主技能获取（ASA）和自适应上下文压缩（ACC）以及两阶段智能体强化学习，实现自我进化。

## 2. 方法论：核心思想、关键技术细节
### 核心思想
- **层次化多智能体架构**：将数据科学工作流分解为多个专门子任务，由Manager Agent负责高层推理、任务分解和子智能体协调；下层包含Cleaner、Featurizer、Modeler、Visualizer、Debugger等子智能体，各自维护局部的技能空间和局部内存，同时共享全局内存（任务目标）。这种设计降低了推理复杂度，并通过将大决策空间分解为结构化子问题，理论上减少了工具选择错误（Theorem 5.1）。
- **自主技能获取（ASA）**：包含四个阶段：
  1. **合成**：当现有技能无法解决子任务时，子智能体通过 prompting 生成新的可执行技能（名称、描述、代码）。
  2. **验证**：执行新技能，仅保留执行成功且输出有效的技能。
  3. **缓存**：将验证通过的技能存入合成技能库（每个子智能体独立）。
  4. **扩展**：采用基于使用频率的策略——当同一技能被重复合成次数超过阈值（τ=3）时，永久加入该子智能体的动作空间。这避免了低质量或罕见技能污染动作空间。
- **自适应上下文压缩（ACC）**：
  - **子智能体层面**：子智能体执行子任务后，基于全局目标将原始执行结果（可能包含大量中间信息）压缩为简洁摘要（成功案例总结关键输出，失败案例总结错误原因）。这有效过滤冗余细节。
  - **Manager Agent层面**：Manager Agent配备专用的摘要工具，可**自适应地**决定何时调用它（而非仅当上下文超限时被动压缩）。压缩时保留任务进度、关键决策和重要中间结果，更新全局内存。
- **两阶段多智能体训练**：
  1. **监督微调（SFT）**：使用高级LLM（DeepSeek-V3.1）作为教师模型收集轨迹（主轨迹和子轨迹），为所有智能体提供稳定初始化。轨迹中包含压缩动作后的分段作为独立样本。
  2. **在线强化学习（RL）**：采用GRPO优化。定义子轨迹奖励（二元：子任务成功0.1，失败-0.1）和主轨迹混合奖励：R_main = R_outcome + αR_sub - βP_context - γP_turn，其中R_outcome是最终解决方案质量，R_sub是子任务平均完成率，P_context和P_turn分别是上下文长度和交互轮次的惩罚项。奖励广播（将最终奖励分配给所有分段轨迹）。总损失包含主轨迹和子轨迹的裁剪代理目标加上KL散度正则化。
- **理论分析**：
  - Theorem 5.1：层次化框架的工具选择错误上界严格低于扁平框架（证明见附录A.3）。
  - Theorem 5.2：Manager Agent的优化目标等价于信息瓶颈问题 min I(Z;C) - λI(Z;Y)，表明ACC能平衡效率与决策质量。

## 3. 实验设计
### 数据集/基准
- **DABench**（257个数据分析问题）：基于52个真实CSV文件，评估端到端数据分析能力。
- **DA-Code**（500个任务）：覆盖数据整理、探索性分析、预测建模等多阶段数据科学流程。
- **ScienceAgentBench（SAB）**（102个任务）：来自4个科学领域的44篇同行评审论文，评估数据驱动科学发现能力。
- **MLE-Dojo**（10个采样实例）：基于真实Kaggle挑战，评估机器学习工程全流程（因计算成本仅采样10个）。
- **可视化任务**：对于DA-Code中的可视化，采用LLM-as-a-judge策略（类似MatPlotBench）。

- **通用智能体**：AutoGen、ReAct、Code Interpreter。

- **骨干模型**：除了DataMind和DeepAnalyze使用固定骨干，其他方法均使用三种LLM骨干：DeepSeek-V3.1-Terminus、GPT-4o、o4-mini。EvoDS使用Qwen3-8B作为共享骨干。
