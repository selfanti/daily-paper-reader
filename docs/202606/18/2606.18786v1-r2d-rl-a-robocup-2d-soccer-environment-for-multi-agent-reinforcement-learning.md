## 摘要
机器人足球是多智能体强化学习的一个具有挑战性的测试平台，因为它结合了部分可观测性、合作与对抗交互、稀疏奖励以及长时域战术行为。RoboCup 2D足球仿真（RCSS2D）提供了一个成熟的机器人足球平台，但其面向竞争的服务器-客户端架构难以直接用于基于Python的现代MARL工作流。我们提出了R2D-RL，一个强化学习环境，通过共享内存通信和周期级同步将RCSS2D和基于HELIOS的玩家客户端连接到Python MARL接口。R2D-RL支持全场和基于场景的训练，具有可配置的对手、基础离散和混合参数化动作空间、动作掩码、基于预期控球值（EPV）的奖励塑形以及并行执行。我们提供了前场射门场景和11对11的全场基准测试，以及基线结果。

## Abstract
Robot soccer is a challenging testbed for multi-agent reinforcement learning because it combines partial observability, cooperative and adversarial interaction, sparse rewards, and long-horizon tactical behavior. RoboCup 2D Soccer Simulation (RCSS2D) provides a mature robot-soccer platform, but its competition-oriented server-client architecture is difficult to use directly with modern Python-based MARL workflows. We introduce R2D-RL, a reinforcement learning environment that connects RCSS2D and HELIOS-based player clients to a Python MARL interface through shared-memory communication and cycle-level synchronization. R2D-RL supports full-field and scenario-based training with configurable opponents, Base discrete and Hybrid parameterized action spaces, action masks, expected possession value (EPV)-based reward shaping, and parallel execution. We provide front-goal scenarios and an 11-vs-11 full-field benchmark, together with baseline results.

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：在 RCSS2D 原生竞争工作流之上添加一个 Python 环境层，通过共享内存和序列计数器协议将服务器、修改后的 HELIOS 客户端、教练和训练器组织成 step 同步的 RL 交互。仅在 `play_on` 模式下暴露策略决策步骤，非 `play_on` 模式（如开球、任意球）由 RCSS2D 规则和 HELIOS 回退行为处理。
- **关键技术细节**：
  - **同步协议**：每个受控玩家维护 `obs_seq`（观测发布序列）、`act_seq`（动作分配序列）、`done_seq`（动作消费序列）。环境等待所有玩家满足 `play_on` 且 `obs_seq > act_seq` 后读取观测和掩码，算法返回联合动作后写入各玩家缓冲区并设置 `act_seq = obs_seq`。玩家消费动作后更新 `done_seq`，环境等待 `done_seq = act_seq` 确认执行。这种分离保证了观测、动作、转换的对应关系。
  - **状态与观测**：
    - 集中状态（教练侧）：136 维向量（球 x/y 位置和速度 + 22 个球员槽位各 6 个特征：x/y 位置、x/y 速度、身体方向、队伍标识）。
    - 局部观测（球员侧）：97 维向量（自身位置/速度/耐力/可踢标志 + 球位置/速度 + 最多 11 个对手和 10 个队友的位置/速度 + 游戏模式/边/门将状态指示器，缺失槽位补零）。
  - **动作空间**：
    - **Base 离散空间**：19 个高层面动作（如传球、射门、带球方向、移动方向、拦截、铲球、接球、HELIOS 回退、空动作），由 HELIOS 行为模块内部决定低层命令和参数。
    - **Hybrid 参数化空间**：6 个离散命令（turn, dash, kick, catch, HELIOS Fallback, empty）各带连续参数，例如 kick 含 power [0,100] 和 angle [-180°,180°]。
    - 两种空间均提供动作掩码：Base 掩码基于 HELIOS 可执行性检查（如射门满足时屏蔽其他动作）；Hybrid 掩码依赖低层命令可行性（如 kick 仅当可踢时可用，catch 仅门将满足条件时可用）。
  - **奖励**：
    - 团队级得分奖励：+1 进球，-1 失球。
    - 可选 MaxEPV 塑形奖励：预计算 32×50 预期控球价值网格，仅当受控队球员可踢时更新最大值 $m_t$，奖励为 $\lambda \max(0, m_t - m_{t-1})$（$\lambda=2$），进球时额外加 $\lambda(v_{\max} - m_t)$。最后奖励为得分奖励 + 塑形奖励之和。
  - **终止条件**：可配置，包括进球、出界、控球丢失（基于最近球员距离判断）、超时（按环境步计数）。
  - **场景重置**：通过训练器直接重设球员和球的状态（位置、方向、速度），无需重启服务器，经若干预热周期后返回首观测。
  - **内置 AI**：对手和非受控队友使用 HELIOS Fallback 动作。

## 6. 主要结论与发现

- **环境有效**：R2D-RL 能支持稳定并行采样（Base 空间峰值 34.85M 步/天，Hybrid 41.65M 步/天），在简单场景（Empty Goal）下所有算法均能近乎满分。
- **动作掩码的重要性**：在 Blocked Shot 中，QMIX 无掩码时进球率为 0%，有掩码时 >87%；MAPPO 对掩码依赖较小（均 >66%）。这表明掩码能帮助算法避免无效/低效动作探索。
- **EPV 塑形帮助复杂任务**：在多攻击者场景中，无 EPV 和无掩码的设置几乎总是零进球率；MaxEPV+mask 组合通常获得最好或唯一正结果（如 Compact Defense 中 MAPPO 仅此组合达到 0.32±0.09）。
- **算法比较趋势**：
  - MAPPO 在大多数场景下最稳定，进球率最高（包括全场）。
  - QMIX 在某些场景表现好但对掩码和种子敏感，全场呈现较大方差。
  - ParaDQN 仅能在单攻击者简单场景成功，多攻击者和全场几乎无学习（甚至无球推进），表明 Hybrid 动作空间联合学习离散命令和连续参数在复杂任务中存在困难。
- **全场局限性**：即使 MAPPO 取得最高进球差（-10.61±1.64），仍明显负值，说明目前算法未能有效战胜 HELIOS 对手。qualitative 示例展示了基本传球和球推进，但无球队员移动固定、缺乏整体阵型。

## 7. 优点

- **环境设计完整**：将成熟竞赛平台转化为标准 MARL 环境，提供集中状态、局部观测、动作掩码、多种动作空间、EPV 塑形、可配置终止和场景重置，便于研究者直接使用。
- **同步机制稳健**：共享内存序列计数器协议保证了每个环境步与服务器周期的严格对应，避免动作丢失或时序错乱。
- **渐进式基准**：从 1v0 到 11v11 的渐进任务设计，有助于分阶段评估算法能力。
- **高吞吐量**：并行采样高效，11v11 下 14 个实例可达 41.65M 步/天，利于大规模训练。
- **开源**：面向未来研究（论文提及开源但未在正文给出链接，根据惯例可预期开放）。
- **清晰定义动作掩码**：Base 和 Hybrid 掩码基于 HELIOS 可执行性检查，减少了无效动作的探索负担。

## 8. 不足与局限

- **对手多样性不足**：所有基准对手均为固定 HELIOS Fallback 行为，限制了策略泛化性和战术多样性。缺乏自对弈、对手池或学习型对手。
- **训练预算有限**：全场仅 30M 环境步，可能远未收敛。论文认可这一点，但未展示更长训练的可能性。
- **非 play_on 模式未覆盖**：开球、任意球、界外球等由规则和回退行为处理，无法学习这些重要战术环节。
- **动作空间人工设计**：Base 高层面动作依赖 HELIOS 内部分配逻辑，引入归纳偏差；Hybrid 动作参数范围选择可能不够优化。
- **ParaDQN 对比不公平**：作为独立学习器，无集中批评且无针对性的混合动作算法优化（如参数化动作特定探索），表现差难以归因于环境限制。
- **环境扩展性限制**：并行超过 14 个实例后吞吐量不再提升，CPU 资源饱和成为瓶颈。未测试 GPU 加速或更高并行度方案。
