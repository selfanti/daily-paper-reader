## 摘要
LLM多智能体系统通常通过自然语言对话或松散结构的共享内存进行协调，使得中间状态难以验证、归因和审计。我们提出PatchBoard，一种基于模式的协作架构，用经过验证的JSON Patch突变替代智能体间对话，在共享结构化状态上操作。一个架构智能体构建任务特定的模式和工作流规则，而一个确定性内核在事务性地提交每个提议的状态突变之前，根据模式约束、角色特定的写入契约和运行时不变式对其进行验证。在630个匹配的ALFWorld场景中，PatchBoard实现了84.6%的成功率，而LangGraph为30.8%，Flock为61.6%；同时每个成功任务的token消耗降至45.5k，而后者分别为368.3k和64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

# 论文详细总结

## 1. 核心问题与整体含义（研究动机和背景）

现有LLM多智能体系统通常通过**自然语言对话**或**松散结构化的共享内存**进行协调。这种方式存在三个关键缺陷：
- **中间状态难以验证**：对话历史混入任务事实、元讨论和修复尝试，导致下游智能体可能将未经验证的观察、过时的计划或错误中间结果当作可靠状态。
- **归因困难**：无法明确区分“模型建议”与“已提交的系统状态”，错误一旦进入共享上下文会被后续智能体放大。
- **审计不足**：缺乏对状态变更的完整、可重放的记录。

上述问题使得长期、有状态的多智能体协作变得脆弱，污染的中介状态会静默影响所有后续智能体。现有方法（如工作流图、黑板系统、结构化输出）虽有所改进，但未能在运行时层面定义“模型提议”与“提交状态”之间的系统级边界。

## 2. 方法论：核心思想、关键技术细节、算法流程

### 核心思想
PatchBoard 将多智能体协作建模为**基于模式的、经过验证的状态突变序列**。用受限的 **JSON Patch** 接口替代自由文本对话，所有状态更新必须通过确定性内核的验证才能进入共享状态。

### 关键技术细节

1. **架构智能体（Architect）**  
   - 在任务初始化时一次性生成**任务蓝图（Blueprint）**，包含：全局状态模式 `Σ`、智能体规范（角色指令、读写路径、上下文预算）、工作流规则 `R`。
   - 蓝图必须通过元模式 `Σ_meta` 验证（格式、路径合法性等），无效蓝图在运行时前即被拒绝。

2. **确定性内核（Deterministic Kernel）**  
   - 维护全局状态树 `St`、事件队列、智能体预算和事务日志。
   - 对每个智能体提议的 JSON Patch `Δ` 执行**五阶段验证**：
     ```
     Accept = Syntax(Δ) ∧ Auth(Δ, Wa) ∧ (Apply(St, Δ) ≠ ⊥) ∧ Valid_Σ(Ŝ) ∧ Inv_B(St, Δ, Ŝ)
     ```
     包括：语法检查、路径授权（角色特定写合约）、临时应用、后状态模式验证、运行时不变式检查。
   - 仅当所有检查通过后才**事务性提交**，拒绝的patch被记录但**不污染共享状态**。
   - 调度器根据已提交状态的事件触发后续智能体调用，使执行流**取决于已验证的状态变更**而非自由消息。

3. **受限上下文视图（Budgeted Context Views）**  
   - 内核为每个智能体构建**有界视图**，仅包含角色允许读取的路径和必要上下文（如未解决依赖、最近拒绝反馈）。
   - 大型集合以摘要和句柄形式呈现，超预算时优先提供最近变更的活跃字段。

4. **结构性属性**  
   - **模式保持**：提交状态始终满足 `Σ`。
   - **角色隔离**：路径级写合约防止一个角色静默覆盖另一角色的输出。
   - **可归因性**：每个接受更新记录智能体ID、视图哈希、patch及验证结果。
   - **可重放性**：事务日志允许不重新采样即可重建状态序列。

### 算法流程（文字说明）
1. 接收用户请求，Architect生成蓝图 → 内核验证蓝图（元模式检查）→ 若无效则拒绝。
2. 内核初始化全局状态，根据工作流规则加入初始事件。
3. 循环：从事件队列取出 `(智能体, 事件)` → 内核切片构造视图 → 调用智能体获取候选patch → 内核执行五阶段验证 → 如果通过则提交状态、记录事务、生成新事件；否则记录拒绝原因且状态不变 → 检查回路策略（连续无效patch、状态循环等）→ 可能触发修复或终止。
4. 返回最终状态和事务日志。

## 6. 主要结论与发现

1. **主结果**：PatchBoard在ALFWorld上成功率**84.6%**（533/630），远高于LangGraph的**30.8%**（194/630）和Flock的**61.6%**（388/630）；且每成功任务仅消耗**45.5k token**，远低于LangGraph的368.3k和Flock的64.2k。
2. **黑板控制**：去除验证和写合约后（即使使用结构化JSON），成功率降至77%，成本上升，证明增益来自验证机制而非仅共享内存。
3. **消融关键组件**：去除patch/schema接口和上下文切片对成功率影响最大（各降15%），去除断路器、Architect、模式强制、写合约均有较小但一致的下降（7-8%）。
4. **灵敏度**：**最小上下文预算（1k字符）** 表现最优，更大预算未提升成功反而增加成本；**生成的模式优于固定模式**，说明Architect的设计有效。
5. **故障隔离**：PatchBoard对无效JSON、路径/类型错误、未授权写入的污染率为**0**（200次注入无一次污染），虚假声明污染率43%（语义无法完全避免），循环终止率**96%**。
6. **边界澄清**：HotpotQA显示所有系统答案准确率相近，但PatchBoard的无效状态率最低（结构有效），而虚假声明率仍存在，证明**模式验证不保证事实正确性**。

## 7. 优点（方法与实验设计亮点）

- **创新架构**：将多智能体协作规范化为“提议-验证-提交”的确定性闭环，首次在系统级分离模型生成与可靠状态。
- **强可靠性保证**：五阶段验证确保只有语法正确、授权、模式兼容、不变式满足的更新才能进入共享状态，从根本上防止污染。
- **完整审计链**：每个状态变更都有明确写入者、提交的视图、patch内容、验证结果，支持重放和调试。
- **资源高效**：通过受限上下文视图和预算控制，有效降低token消耗，避免无关信息干扰。
- **实验设计严谨**：多维度对比（基础方法、控制变体、消融、灵敏度、故障注入、语义诊断），统计推断使用置信区间，结论可靠。
- **明确局限性声明**：不宣传语义保证，在HotpotQA中主动展示虚假声明率，诚实区分结构有效性与事实正确性。

## 8. 不足与局限

- **不保证语义正确性**：模式验证无法阻止虚假或未支持的声明，事实性任务仍需额外证据检索和验证器。
- **依赖蓝图质量**：过于稀疏或过于宽泛的模式会损害效果；Architect能力直接影响系统上限（实验中生成模式优于固定模式，但非最优保证）。
- **场景覆盖有限**：最强证据来自ALFWorld（任务状态和成功条件明确），更开放的场景（软件工程、科学发现、长文本生成）可能需要更复杂的模式和验证逻辑。
- **工程开销**：需要设计模式、上下文切片、patch验证、事务日志等组件，适合对可审计性和状态完整性要求高的场景，简单任务可能过度设计。
- **故障注入揭示的语义缺口**：虚假声明的污染率仍达43%（在HotpotQA中），说明即使结构正确的更新也可能传播错误信息，需要更强的语义检查。
- **可扩展性担忧**：当智能体数量或状态树规模很大时，确定性内核的验证和切片可能成为瓶颈（论文未评估大规模部署）。
- **模型依赖**：实验仅使用一种模型（Qwen-plus），其他模型（如GPT-4、Llama）的表现需进一步验证。
