## 摘要
一个自演化智能体循环不断提出智能体的修改版（其提示模板或程序），并基于每次迭代的质量信号接受或拒绝该修改。设计该信号通常是项目中最昂贵的部分：可靠的标量奖励需要领域专业知识和标记样本，而这些样本的组装成本与智能体底层任务本身一样高。我们建议用成对验证器替换接受/拒绝门处的标量：一个冻结的LLM，给定父代和子代候选，返回哪个更好的二元判定。成对判断通常比绝对评分更容易且更稳定，因为其对比性质减少了对严格尺度校准的需求。该验证器也无需自身训练。我们将验证器集成到三个已发表的自演化引擎（GEPA、ADRS、ShinkaEvolve）中，并报告两种变体：自适应聚焦（保留引擎现有的验证集父代选择）和软Elo（让验证器的判定驱动父代选择，从而也降低验证集奖励）。跨多个智能体和两种人工制品底层（提示和代码），我们的方法在我们评估的大多数设置上匹配或超过完全奖励基线，并且该模式在跨族验证器交换中依然成立。因此，成对门是每步奖励设计的即插即用替代方案，具有竞争力的任务准确性且无需标注成本。

## Abstract
A self-evolving agentic loop repeatedly proposes a tweaked version of an agent (its prompt template or program) and accepts or rejects the change based on a per-iteration quality signal. Designing that signal is often the costly part of the project: a reliable scalar reward requires domain expertise and labeled examples that are themselves as expensive to assemble as the agent's underlying task. We propose replacing the scalar at the accept/reject gate with a pairwise validator: a frozen LLM that, given the parent and child candidate, returns a binary verdict on which is better. Pairwise judgment is generally easier and more stable than absolute scoring, due to its contrastive nature, which mitigates the need for strict scale calibration. The validator also requires no training of its own. We integrate the validator into three published self-evolving engines (GEPA, ADRS, ShinkaEvolve) and report two flavors: Adaptive Focus, which retains the engine's existing val-set parent selection, and Soft Elo, which lets the validator's verdicts drive parent selection so that val-set rewards drop as well. Across multiple agents and two artifact substrates (prompt and code), our method matches or exceeds the full-reward baseline on the majority of settings we evaluate, and the pattern survives a cross-family validator swap. The pairwise gate is thus a drop-in replacement for per-step reward design at competitive task accuracy without the labeling cost.

## 1. 论文的核心问题与整体含义（研究动机和背景）

自我进化智能体循环（self-evolving agentic loop）通过反复提出智能体的修改版（如提示模板或程序），并基于每步质量信号决定是否接受修改。然而，设计可靠的标量奖励信号是项目中最昂贵的部分：需要领域专家、大量干净标注样本，其成本往往与智能体底层任务本身相当。本文的核心问题是如何在不使用标量任务奖励的情况下，仍能驱动智能体有效进化。

整体含义：提出用**成对验证器（pairwise validator）**——一个冻结的 LLM，给定父代和子代候选后输出二元“哪个更好”的判定——替代每步的标量奖励。由于成对比较天然具有对比性，比绝对评分更稳定、更少受尺度校准影响，且无需额外训练。该替换有望大幅降低标注成本，同时保持任务准确度。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- 将进化循环的接受/拒绝门（gate）从“子代得分>父代得分”替换为验证器 V<sub>ϕ</sub> 的二元输出：V<sub>ϕ</sub>(ft,c, ft,p, E) 返回 “better” 或 “worse”。
- 验证器是**冻结的 LLM**，无需训练，仅需少量上下文示例。
- 验证器在每次迭代中使用一个训练小批量 T<sub>t</sub>（|T<sub>t</sub>|=3）进行判断，以多数投票决定接受与否。

### 关键技术细节
- 提出两个变体：
  1. **Adaptive Focus（适应聚焦）**：只替换门，保留引擎原有的验证集父选择。验证器在三种预定义评价维度（Accuracy、Soundness、Holistic）间自适应轮换，每 10 迭代由元 LLM 刷新。
  2. **Soft Elo（软 Elo）**：完全替换父选择，使用 Elo 评分系统。每个候选有初始评分 1600；每次迭代，子代与父代及 3 个随机存档成员进行匹配，根据验证器判定结果更新 Elo 评分（公式 4），权重引入智能体 token 置信度 c<sub>t</sub>。父代通过 softmax 采样。

- 验证器上下文 E 的设计：有 Direct（无额外）、Few-shot（加 3 个示例，带或不带奖励标签）、Adaptive Focus（运行时变化的准则）等变体。

### 公式流程
- 传统门：a<sub>t</sub>ᶠᵘˡˡ = 1{r(ft,c) > r(ft,p)}
- 验证器门：a<sub>t</sub>^V = 1{V<sub>ϕ</sub>(ft,c, ft,p, E) = better}
- Elo 更新：R<sub>c</sub> ← R<sub>c</sub> + k·c<sub>t</sub>·(S(x) − Ŝ<sub>c</sub>)

## 6. 论文的主要结论与发现

- 成对验证器可替代每步标量奖励，**在多数设置（20/20 提示进化行、5/10 代码进化单元）匹配或超过** 完全奖励基线。
- **跨验证器族**（将 Qwen3-8B 换成 Claude Haiku 或 Gemma-4）表现稳定，提示增益不是自我配对伪影。
- 验证器门减小了验证-测试差距（当验证集小且噪声大时），因不依赖验证集评分作为选择信号。
- Soft Elo 在噪声验证集场景（如 HoVer）更优；Adaptive Focus 在指令遵循任务（如 IFBench）更优。
- 代码进化中，**Direct + full reward** 表现最佳，说明替换门而非父选择是主要贡献。

## 7. 优点

1. **无需训练**：验证器为冻结 LLM，零训练成本，即插即用于现有引擎。
2. **降低标注成本**：仅需少量成对示例（3 个），无需大规模标量奖励数据。
3. **对比稳定性**：成对比较比绝对打分更可靠，缓解标量校准问题。
4. **跨模态泛化**：在 prompt 和 code 两种基质上均有效。
5. **增量可选**：提供从部分奖励自由（Adaptive Focus）到完全奖励自由（Soft Elo）的平滑过渡。
6. **跨模型族泛化**：验证器可来自不同模型族（甚至更小模型），无自归因偏差。

## 8. 不足与局限

1. **敏感区域**：四种失败模式——基线饱和、小测试集（AIME 仅 15 题，单个问题改变 6.7%）、智能体能力不足、验证器自身能力不匹配（Gemma-4 在 PUPA 上表现下降）。
2. **实验覆盖**：仅测试了 3 个引擎，未覆盖全部现有自进化框架（如 AlphaEvolve、AFlow）。代码任务数量较少。
3. **成本增加**：验证器门增加了 LLM 调用次数（1.7~2.4 倍），尽管每次调用极短（1 token 输出），但在大规模部署中可能显著增加延迟和 API 费用。
4. **缺乏理论分析**：未严格证明为何成对比较泛化性更好，仅基于经验观察。
5. **未与多目标进化比较**：作者承认应进一步与直接把验证器评分作为额外目标的多目标 Pareto 方法对比。
6. **确定性限制**：验证器使用贪心解码，可能遗漏更具探索性的接受决策；Soft Elo 的 Elo 步骤 size 需要扫描调优。
