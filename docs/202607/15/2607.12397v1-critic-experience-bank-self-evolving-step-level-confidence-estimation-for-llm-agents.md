## 摘要
LLM智能体在外部环境中行动，每个动作都会改变后续决策所依赖的状态，并且一个错误的步骤可能在观察到最终失败之前就浪费交互预算或引发不可逆的副作用。因此，可靠的部署需要步骤级的置信度估计：在动作执行之前，每个提议动作是否富有成效的校准概率。现有的LLM置信度估计器设计用于根据给定提示对回答进行评分，但智能体的置信度还取决于执行后果：相似情境下的类似动作在环境响应后是否真正推进了任务。我们提出了\method（\methodshort），一个自演化的批评者框架，其中LLM批评者从其自身过去的判断及其观察到的后果中积累证据。在每个轨迹之后，一个看到完整执行反馈的后见之明LLM对每个步骤是否富有成效进行投票。由此产生的伪标签填充一个记忆库，当类似步骤再次出现时，从该库中检索相关的富有成效和非富有成效的经验到批评者的提示中。\methodshort不需要训练，也不使用真实步骤标签。在三个智能体基准和三个批评者骨干网络上，\methodshort在每个数据集-批评者组合上都取得了最佳的校准（ECE和Brier）和排序（AUC），相对于最强的无训练基线，ECE降低了高达54%。

## Abstract
LLM agents act in external environments where each action changes the state that later decisions condition on, and where a single wrong step can waste interaction budget or trigger irreversible side effects long before the final failure is observed. Reliable deployment therefore requires \emph{step-level confidence estimation}: a calibrated probability that each proposed action is productive, available \emph{before} the action is executed. Existing LLM confidence estimators are designed to score a response from the given prompt, but agent confidence also depends on execution consequences: whether similar actions in similar situations actually advanced the task after the environment responded. We introduce the \method (\methodshort), a self-evolving critic framework in which an LLM critic accumulates evidence from its own past judgments and their observed consequences. After each trajectory, a hindsight LLM that sees the full execution feedback votes on whether each step was productive. The resulting pseudo-labels populate a memory bank from which related productive and unproductive experiences are retrieved into the critic's prompt whenever a similar step recurs. \methodshort requires no training and uses no ground truth step labels. Across three agent benchmarks and three critic backbones, \methodshort attains the best calibration (ECE and Brier) and ranking (AUC) in every dataset--critic combination, reducing ECE by up to $54\%$ relative to the strongest training-free baseline.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：LLM 智能体（如网页导航、移动 GUI、Shell 交互）在执行任务时，每一步动作都会改变环境状态，且单个错误步骤可能浪费交互预算或引发不可逆的后果（如提交表单、删除文件）。因此，在动作执行前需要**步骤级（step-level）置信度估计**：对每个提议动作是否富有成效进行校准的概率估计。
- **现有方法的不足**：已有的 LLM 置信度估计方法（如 token logits、一致性、口头化置信度）是为单次响应设计的，只考虑当前提示，忽略了执行后果——即相似动作在相似状态下的实际结果。过程奖励模型（PRM）和步骤级验证器虽有效，但需要标注的步骤数据和可训练模型，不适合部署场景（固定 API 模型、无步骤标注）。
- **整体含义**：提出一种无需训练、无需真实步骤标签、自演化的批评者框架 CEB，通过检索自身过去的执行经验来校准置信度，填补了现有方法在智能体部署中的关键空白。

## 2. 方法论

- **核心思想**：利用已完成轨迹的执行反馈，通过后见之明（hindsight）LLM 为每个步骤生成伪标签（是否富有成效），构建一个经验库（Experience Bank）。当新步骤出现时，检索与之相似的历史经验（包含正反案例）放入批评者的 prompt 中，从而在动作执行前获得更可靠的置信度。
- **关键技术细节**：
  - **轨迹表示**：$\tau = (s_1, a_1, o_1), \dots, (s_T, a_T, o_T)$，其中 $s_t$ 是执行前的状态，$a_t$ 是提议动作，$o_t$ 是执行后的观察。
  - **置信度估计**：$\hat{z}_t = f(q, s_t, h_{<t}, a_t)$，其中 $f$ 是固定 LLM 批评者，但通过检索增强（上下文中加入历史经验）。
  - **经验库构建**：每个轨迹完成后，后见之明 LLM $g_\phi$ 在完整轨迹条件下进行 $V$ 次独立采样（温度 $\tau>0$），通过多数投票得到每个步骤的伪标签 $\hat{y}_t$。每条记录包含：任务 $q'$、状态摘要 $s'$、动作 $a'$、观察摘要 $\tilde{o}'$、伪标签 $\hat{y}'$、历史置信度 $\hat{z}'$ 以及一致性标志 $\chi'$（记录批评者原始判断与后见之明结果是否一致）。
  - **检索和提示**：新步骤形成查询 $\kappa_t = \text{task} \| \text{state} \| \text{action}$，使用 TF-IDF 余弦相似度从经验库中检索 top-k 正例和 top-k 反例（若某类不足则全部取出），并加入重复检测标志 $\rho_t$（指示当前动作是否与最近几步重复）。批评者 prompt 包含任务、当前状态、历史、提议动作、检索到的正反示例以及重复标志，输出一个数值分数 $\hat{z}_t$（clip 到 [0,1]）。
- **算法流程**（文字说明）：初始化空库；对每个任务，循环步骤直到完成：对当前动作形成复合查询，检索 top-k 正反例，调用批评者得到置信度，执行动作并获得观察；轨迹结束后，后见之明 LLM 投票产生伪标签，将每步记录加入经验库。

## 6. 主要结论与发现

- CEB 在所有 9 个数据集-骨干组合中取得最佳 ECE、Brier 和 AUC，ECE 相对最强无训练基线降低最多 54%（InterCode-Bash 上 GPT-4.1）。

- 消融实验表明，**正反对比检索**是最大贡献因素（去除后 ECE 增加 +0.065），其次是**基于 TF-IDF 的复合键检索**（+0.042），投票降噪和重复信号提供补充。
- 自演化行为：随着经验库增长，校准误差持续下降，而静态批评者无明显变化。
- 选择性执行实验：CEB 能最好地将错误步骤集中在低置信度区域，保留准确率和任务成功率提升优于所有基线，接近 Oracle。

## 7. 优点

- **无需训练和真实标签**：完全基于冻结模型和自身执行反馈，适合 API 模型部署。
- **步骤级、执行前置信度**：直接解决智能体部署中的实际需求。
- **自演化机制**：批评者上下文随库增长而改进，无需更新权重。
- **对比性检索设计**：同时展示正反案例，帮助批评者校准过度/不足置信。
- **全面实验验证**：覆盖三个不同模态基准、多个骨干、多种指标和附加分析，结论稳健。
- **对伪标签噪声鲁棒**：接近 GT oracle 上限（最高闭合 95% 的校准差距），说明后见之明投票效果良好。

## 8. 不足与局限

- **批评者与后见之明 LLM 相同**：未隔离独立标签器的影响，可能存在偏差。
- **步骤标签依赖参考或 LLM 重标**：在 InterCode-Bash 上奖励派生标签可能有噪声。
- **无库淘汰或压缩机制**：对于超大规模部署流，无限增长的经验库可能导致检索效率下降和 token 成本增加。
- **检索基于 TF-IDF**：可能不如学习型密集检索，但实验中 TF-IDF 反而优于密集检索（附录 F），作者解释为任务匹配更依赖精确词汇而非语义。
- **仅评估线上选择性执行（模拟）**：未在真实闭环 replanning 中进行测试。
- **算力消耗未完整报告**：仅提供了 token 成本，缺少总体推理时间和能耗数据。
