## 摘要
我们提出HORIZON，一个自演化智能体框架，将硬件设计视为仓库级代码演化。一个Markdown工具集被编译成包含领域知识、可执行评估器、接受谓词以及git/运行时策略的项目包；然后一个免干预的智能体循环演化一个隔离的git工作树，使用仓库操作进行状态管理、追踪和重放。这扩展了先前从EDA软件系统到硬件设计工件本身的仓库级自演化工作。我们在ChipBench、RTLLM、Verilog-Eval以及九个CVDP类别上评估了我们的方法，通过完全免干预的智能体循环在所有套件上实现了100%的基准完成率。然而，我们不声称硬件设计的智能体AI问题已解决：这些基准只是更广泛的芯片设计工程问题的受控代理。第\ref{sec:discuss}节分析了当前研究的局限性并指出了开放的研究挑战。

## Abstract
We present HORIZON, a self-evolving agent framework that treats hardware design as repository-level code evolution. A Markdown harness is compiled into a project pack containing domain knowledge, an executable evaluator, an acceptance predicate, and a git/runtime policy; a hands-free agent loop then evolves an isolated git worktree, using repository operations for state management, tracing, and replay. This extends prior works of repository-scale self-evolution from EDA software systems, to hardware-design artifacts themselves. We evaluate our approach on ChipBench, RTLLM, Verilog-Eval, and nine CVDP categories, achieving 100\% benchmark completion across all suites with a fully hands-free agentic loop. However, we do not claim that agentic AI for hardware design is solved: these benchmarks are controlled proxies for a much broader engineering problem in chip design. Section~\ref{sec:discuss} examines the limitations of the current study and highlights open research challenges.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：硬件设计（尤其是RTL设计）自动化面临一个关键瓶颈：单次代码生成无法满足复杂、迭代式的设计任务。设计必须经过编译、仿真、波形检查等过程，而现有LLM生成的“看起来合理”的Verilog往往无法通过验证。作者希望将硬件设计问题转化为**仓库级代码演化**问题——即在一个隔离的git工作树中，通过可执行的反馈循环（编译、仿真、接受谓词）自主迭代改进设计，直到满足接受条件。
- **背景**：此前的工作（AlphaEvolve、SATLUTION、ABCEvo）展示了LLM结合自动评估器能够在软件层面（如算法内核、SAT求解器、EDA工具）进行仓库级自演化。但硬件设计本身尚未被纳入这一范式。本文提出HORIZON，将同一原则应用到硬件设计工件（RTL源文件、测试平台等），使设计问题变成一个自包含的git工作树演化过程，从而减少人工干预并实现可重复的自动化设计修复。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将硬件设计任务封装为**版本控制的仓库演化问题**。用户提供一个结构化的Markdown工具集（harness），包含目标、领域知识、评估器规范和接受谓词；一个**引导智能体**（bootstrap agent）将其编译成**项目包**（project pack），包含代理策略提示、可执行评估器、接受谓词、git/运行时策略和领域技能；随后一个**免干预智能体循环**自动迭代编辑工作树、运行评估器、根据接受谓词决定提交或拒绝，并利用原生git操作记录每个版本的状态、差异、提交信息和评估证据。
- **关键技术细节**：
  - **形式化定义**：引入半马尔可夫决策过程（semi-MDP）的结构化命名（非行为假设）。状态 \(s_t\) 包含当前git树、项目包、战役状态、累积日志、声明的记忆；选项 \(a_t\) 包含建议的补丁、工具调用序列、最终审查决策；评估器产生证据 \(y_t\)；接受谓词决定是否提交新版本。
  - **执行轨迹**：轨迹 \(\tau_{0:D}\) 记录了每个外循环步骤的状态、动作、奖励、下一状态和证据，可用于策略分析、奖励建模、课程学习或离线强化学习（但本文中不进行RL训练）。
  - **会话复用与缓存**：通过持久化模型会话，使工具集、项目包、稳定源文件等从提供商的提示缓存中服务，而不是每次重新发送。新计费token主要来自当前差异、最新评估器输出和代理响应，使得额外修复迭代的边际成本很低。
  - **Git作为追踪基底**：git diff查看暂存的编辑，git commit记录接受版本并附加评估结果，git log恢复完整历史。提交记录成为正面例子，拒绝记录成为负面例子，仓库历史本身即经验缓冲区。

## 6. 论文的主要结论与发现

- **主要发现**：HORIZON在所有评估基准上均达到**100%完成率**。仅有一个ChipBench任务因原始基准中的规格-工具集不匹配而失败。
- **收敛速度差异**：简单基准（RTLLM、Verilog-Eval）在2次迭代内达到100%；CVDP类别需要更多迭代，其中CID 002（RTL代码补全）最困难，需82次迭代。
- **Token消耗集中**：总token约210M，其中97%消耗在CVDP类别；CID 002、003、012占比最高（分别为26.7%、18.1%、15.3%）。约91%的token为缓存输入，大幅降低了API成本。
- **覆盖率观察**：在CID 012（测试向量生成）中，通过率达到100%时平均覆盖率仅为97.9%，说明接受谓词仅要求通过特定测试，而非最大化覆盖率。
- **关键观点**：当前RTL基准无法检测**过求解（over-solving）**或**奖励黑客**风险，社区需要像SWE-bench那样分离诊断反馈与最终评分。

## 7. 优点

1. **方法创新性**：首次将仓库级自演化从EDA软件扩展到硬件设计工件本身，利用git原生操作实现状态管理、追踪和重放，形成统一的框架。
2. **完全免干预**：一旦用户提供harness，智能体循环无需人类干预即可运行至收敛，展示了自主设计修复的可行性。
3. **广泛的基准覆盖**：覆盖了生成、修改、验证、调试等多样化任务类型，证明了框架的通用性。
4. **细致的token分析**：报告了累计token消耗及缓存比例，为用户提供了实际部署成本估算依据。
5. **坦率的限制讨论**：作者明确指出现有基准的局限性（过求解风险、反馈延迟等），并为未来研究工作设定了方向。

## 8. 不足与局限

1. **实验对比缺失**：未与任何现有迭代式RTL生成系统（如AutoChip、ACE-RTL、VerilogCoder）进行对比，因此无法量化HORIZON相对于现有方法的优势。
2. **缺乏统计可靠性**：单次运行未提供方差或置信区间，结果可能受随机种子影响。
3. **没有消融分析**：无法确定git追踪、会话缓存、提示结构等组件各自的贡献。
4. **奖励黑客风险**：当前评估器暴露全部诊断反馈，智能体可能针对特定测试用例“过度拟合”，而非实现真正鲁棒的设计。
5. **基准本身局限**：所有基准都是受控代理，不能代表真实芯片设计中的不完整规格、PPA权衡、下游集成等。

7. **未进行RL训练**：虽然形式化为半MDP，但智能体策略固定为GPT-5.3，无法从历史轨迹中学习改进演化规则。
