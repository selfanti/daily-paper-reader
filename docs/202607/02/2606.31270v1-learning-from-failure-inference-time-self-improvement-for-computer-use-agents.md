## 摘要
计算机使用智能体利用多模态大语言模型（MLLMs）操作计算机并完成任务，因其实用性和多功能性而受到广泛关注。开发这些智能体的一个主要挑战是收集大规模、高质量的轨迹。标准方法通过自我改进循环生成合成数据：将智能体置于可验证环境中，并对其成功轨迹进行迭代微调。尽管该方法有效，但它仅利用成功轨迹而丢弃失败轨迹，尽管失败中包含了关于模型弱点的丰富信息。在这项工作中，我们探索了一种互补的失败驱动自我改进循环，这是一种以数据为中心的范式，将失败轨迹转化为智能体改进。具体而言，我们利用LLM诊断失败模式，提出推理时解决方案，并生成代码补丁——经人类轻量验证——以升级智能体。我们在OSWorld基准上使用最先进的OpenCUA-72B模型验证了该方法，将成功率从42.3%提升至48.9%，提升了6.6个百分点，且无需额外训练成本，仅增加少量推理开销。我们的结果表明，失败驱动的自我改进是基于成功的流程的有效补充，能够实现更高效的智能体改进。

## Abstract
Computer-use agents, which leverage multimodal large language models (MLLMs) to operate computers and complete tasks, have attracted significant attention for their utility and versatility. A major challenge in developing these agents is collecting large-scale, high-quality trajectories. The standard approach generates synthetic data through a self-improving loop: an agent is placed in a verifiable environment and iteratively fine-tuned on its successful trajectories. Despite its effectiveness, this paradigm exploits only successful trajectories and discards the failed ones, even though failures carry rich information about a model's weaknesses. In this work, we explore a complementary failure-driven self-improvement loop, a data-centric paradigm that turns failed trajectories into agent improvements. Specifically, we employ an LLM to diagnose failure modes, propose inference-time solutions, and generate code patches -- lightly verified by humans -- that upgrade the agent. We validate this approach with the state-of-the-art OpenCUA-72B model on the OSWorld benchmark, improving the success rate from 42.3% to 48.9%, a gain of 6.6 percentage points, without any additional training cost and with only modest inference overhead. Our results demonstrate that failure-driven self-improvement is a viable complement to success-based pipelines, enabling more efficient agent improvement.

## 1. 论文的核心问题与整体含义（研究动机和背景）

-   **核心问题**：当前计算机使用智能体（Computer-Use Agents）的主流自我改进方法（成功-驱动循环）只利用成功轨迹对模型进行微调，而**完全丢弃了包含丰富信息的大量失败轨迹**，导致数据利用效率低下。
-   **背景**：构建高性能计算机使用智能体的关键在于获取高质量的操作轨迹（Trajectories）。由于人工标注成本高且难以规模化，现有工作如OpenCUA、UI-TARS等通常采用“成功-驱动”循环：在可验证环境中执行任务，自动筛选出成功轨迹，然后使用这些轨迹进行监督微调（SFT）。这一范式虽有效，但浪费了在构建可验证环境时产生的绝大多数失败经验。

## 2. 论文提出的方法论：核心思想、关键技术细节

-   **核心思想**：提出一种新颖的、与成功-驱动循环互补的**失败-驱动自我改进循环**。该循环将失败轨迹转化为推理时的行为改进，通过利用大语言模型（LLM）诊断失败原因并生成代码补丁来实现智能体升级，且无需额外训练。
-   **关键技术细节（失败-驱动循环流程）**：
    1.  **失败经验收集**：在可验证环境中执行任务，收集所有失败轨迹（含指令、行动历史、思考过程）。
    2.  **LLM引导的诊断与改进**：将失败轨迹输入LLM（作为元控制器，文中实验选用Claude 4.5 Sonnet），LLM分析并归纳出四类核心失败模式（见下文），随后为每类模式提出对应的推理时优化策略（即代码补丁）。
    3.  **人类轻量验证**：人类对LLM生成的代码补丁进行最小化审查和调整（<3%的修改行数），确保其正确性。
    4.  **应用与迭代**：将补丁集成到智能体的工作流程中，重新进行评估，开启新一轮的失败收集与改进循环。策略会逐步累积。
-   **识别的四种失败模式与解决方案**：
    -   **定位错误 (Grounding Errors)** → **视觉搜索 (Visual Search)**：当智能体执行点击、移动等操作后，提取目标区域的放大视图并添加标记，让智能体自我验证坐标是否正确，若错误则进行调整。
    -   **能力差距 (Competency Gaps)** → **终端执行 (Terminal Execution)**：允许智能体通过快捷键打开终端，将复杂的GUI操作转化为更直接、可靠的命令行指令。
    -   **知识缺陷 (Knowledge Deficiencies)** → **知识支持 (Knowledge Support)**：提供两种外部知识源：1）**搜索引擎接口**，允许智能体在遇到不熟悉的命令时可查询预定义的搜索模型（如GPT-5-mini）；2）**软件手册检索器**，针对特定软件（如LibreOffice）提供精简的快捷键参考手册。
    -   **冗余循环 (Redundant Loops)** → **重复警告 (Repetition Warnings)**：设计一个检测模块，监控智能体的**思考过程**、**动作**和**屏幕状态**是否在滑动窗口内重复出现，一旦检测到停滞（如屏幕状态3步不变），则触发恢复模式，提示智能体尝试替代策略。
