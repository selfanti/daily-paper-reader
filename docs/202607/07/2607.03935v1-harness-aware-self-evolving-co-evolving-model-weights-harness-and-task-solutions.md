## 摘要
自演化框架通常优化任务解决方案，同时将周围的框架视为固定不变。我们引入了感知框架的自演化（HASE），这是一种基于智能体强化学习的框架，其中单个模型可以在多轮动作空间中生成任务解决方案或编辑选定的框架组件。HASE使得单个Qwen3-8B模型能够匹配使用Claude Code作为框架提议者的GPT-OSS-120B模型的文本分类性能。在阿尔法因子挖掘中，HASE超越了已报道的GPT-OSS-120B基线。HASE还修复了不完美的评估组件，并在圆形填充算法发现中收敛到最先进的性能。这些结果表明，HASE通过一个统一的智能体过程改进了框架和解决方案。

## Abstract
Self-evolving frameworks usually optimize task solutions while treating the surrounding harness as fixed. We introduce Harness-Aware Self-Evolving (HASE), an agentic reinforcement-learning framework in which a single model can generate task solutions or edit selected harness components in a multi-turn action space. HASE enables a single Qwen3-8B model to match the text-classification performance of a GPT-OSS-120B model that uses Claude Code as the harness proposer. In alpha factor mining, HASE outperforms the reported GPT-OSS-120B baseline. HASE also repairs imperfect evaluation components and converges to state-of-the-art performance in circle-packing algorithm discovery. These results show that HASE improves the harness and the solution through one unified agentic process.

### 论文详细总结

#### 1. 核心问题与整体含义

*   **背景与动机**：当前基于大型语言模型（LLM）的**自演化系统**（如FunSearch、AlphaEvolve等）通常只关注优化任务的“对象级解决方案”（如代码、算法），而将与模型交互的**支撑组件（Harness）**（如提示词、记忆库、检索策略、评估器、工具接口等）视为固定不变。
*   **核心问题**：这种固定支撑组件的假设存在两大局限：
    1.  **评估器不完善**：人工编写的评估器可能不完整或与真实环境错位，模型可能为了高分而利用这些漏洞（奖励黑客），而非真正解决问题。
    2.  **引导组件不佳**：提示词遗漏有效结构、记忆库冗余、检索策略不佳等，会限制模型的搜索能力。
*   **关键洞见**：支撑组件与模型权重同样重要，决定了模型能观察什么、能采取什么行动以及如何被评估。因此，**同时优化支撑组件与解决方案**能更有效地提升系统整体能力。

#### 2. 方法论

*   **核心思想**：提出 **HASE（感知框架的自演化，Harness-Aware Self-Evolving）** 框架，通过**智能体强化学习**，让**同一个模型**可以在一个统一的**多轮动作空间**内，自由选择是生成任务解决方案还是编辑支撑组件，从而实现模型权重、任务方案和支撑组件的协同演化。
*   **关键技术细节**：
    *   **两类支撑组件**：
        1.  **引导组件（Guidance Components）**：包括提示词、记忆格式化、上下文检索等。这些组件可以自由改进，以提升搜索质量。
        2.  **评估组件（Evaluation Components）**：包括验证器和评分器。它们的修改必须与**真实世界评估器（Oracle Evaluator）** 的结果对齐，以防止奖励黑客。
    *   **统一动作空间**：模型可以选择“提交方案”、“检查文件”或“编辑支撑组件文件”。支撑组件的编辑只在当前轨迹内生效，只有经过阶段审查的编辑才会被永久提交。
    *   **两阶段演化**：
        1.  **引导组件演化**：使用**概念验证（PoC）** 分数进行初步筛选，只有能带来性能提升的编辑才会进入候选池。在阶段边界，通过验证集性能对候选池进行审查，选择最佳补丁进行提交。
        2.  **评估组件演化**：当**本地评估器（Proxy）** 的分数与**真实世界评估器（Oracle）** 的分数出现分歧时，将该案例加入**错配集（Mismatch Set）**。评估器补丁必须能修复错配集中的更多案例，同时不引起回归，才会被接受。这确保了评估器的修复是受约束的、目标明确的。
    *   **强化学习驱动**：模型通过GRPO算法进行训练。奖励仅基于最终提交方案的质量，而非编辑支撑组件的行为本身。这鼓励模型为了获得更好的解决方案而进行有意义的支撑组件修改。

*   **算法流程**：
    1.  初始化模型策略和支撑组件。
    2.  进入阶段循环。每个阶段包含多个GRPO训练周期。
    3.  在每个周期内，模型在多轮交互中生成轨迹（生成方案或编辑支撑组件）。
    4.  使用本地评估器对轨迹中的最终方案评分，并通过GRPO更新模型权重。
    5.  在阶段结束时，收集候选支撑组件补丁。对于引导组件，通过验证集审查；对于评估组件，通过错配集审查。
    6.  提交审查通过的支撑组件，进入下一阶段。
