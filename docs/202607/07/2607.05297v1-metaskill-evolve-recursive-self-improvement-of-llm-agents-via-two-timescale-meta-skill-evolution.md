## 摘要
近期大型语言模型（LLM）智能体处理日益长时程、开放式的任务，而外部技能（即提供给智能体的可复用程序性知识）进一步扩展了这种能力。然而，固定的手工编写技能很少是最优的，并且无法适应智能体遇到的任务多样性。自我改进的智能体通过从执行轨迹中重写自己的技能文件来解决这一问题，在具有挑战性的基准测试中取得了显著的性能提升。然而，这种自我进化仍然是非递归的：它只改进了任务技能（智能体做什么），而改进过程（如何改进）是一次性编写并保持固定的。我们提出了MetaSkill-Evolve，一个双时间尺度框架，使智能体技能改进变得递归：每个分支都携带一个任务技能$s$和一个分支局部元技能$m=(ψ,σ,α,π,\varepsilon)$，其五个组件分别参数化了改进流程中的分析器、检索器、分配器、提议器和进化器智能体。任务技能在快速循环上进化，而元技能在较慢循环上通过应用相同的流程对自身进行进化，无需额外的模型或目标。在五个流程智能体共享单个冻结主干的情况下，MetaSkill-Evolve在三个智能体基准测试（OfficeQA、SealQA、ALFWorld）上优于无技能、静态技能和单层进化基线，相对于原始主干在保留测试集准确率上分别提高了+23.54、+16.09和+1.92个百分点。

## Abstract
Recent LLM agents tackle increasingly long-horizon, open-ended tasks, and external skills, reusable procedural knowledge supplied to the agent, further extend this capability. However, a fixed, hand-authored skill is rarely optimal, and cannot adapt to the diversity of tasks an agent encounters. Self-improving agents address this by rewriting their own skill files from execution traces, yielding meaningful gains on challenging benchmarks. Yet such self-evolution remains non-recursive: it improves only the task skill (what the agent does) while the improvement procedure (how it improves) is authored once and held fixed. We introduce MetaSkill-Evolve, a two-timescale framework that makes agentic skill improvement recursive: every branch carries both a task skill $s$ and a branch-local meta-skill $m=(ψ,σ,α,π,\varepsilon)$ whose five components parameterise the Analyzer, Retriever, Allocator, Proposer, and Evolver agents of the improvement pipeline. Task skills evolve on a fast loop while the meta-skill evolves on a slower one under the same pipeline applied to itself, with no additional model or objective. With all five pipeline agents sharing a single frozen backbone, MetaSkill-Evolve outperforms no-skill, static-skill, and single-level evolution baselines on three agentic benchmarks (OfficeQA, SealQA, ALFWorld), improving held-out test accuracy over the raw backbone by +23.54, +16.09, and +1.92 points respectively.

## 1. 核心问题与整体含义（研究动机和背景）

当前LLM智能体在执行长周期、开放式任务时，往往依赖外部“技能”（可复用的过程性知识文件）。但手工编写的固定技能无法适应任务多样性。现有自改进方法（如EvoSkill、GEPA、SkillWeaver）通过“分析-提议-进化”管道从执行轨迹重写技能文件，但**仅优化任务技能本身，而改进过程（如何改进）是固定不变、由开发者一次性编写的**。这种非递归的自我进化限制了智能体适应能力：每个分支使用相同的诊断风格、编辑提案策略等，无法根据失败类型自适应调整。

论文核心研究问题：**能否将改进过程本身作为一等对象，与任务技能一起演化，使用相同的智能体管道进行递归自改进？**

## 2. 方法论：核心思想、关键技术细节

### 2.1 双时间尺度框架

提出**MetaSkill-Evolve**，每个分支状态包含：
- **任务技能 $s$**：Markdown格式的LLM智能体程序。
- **分支局部元技能 $m = (\psi, \sigma, \alpha, \pi, \varepsilon)$**：五个组件分别参数化改进管道的五个阶段：
  - $\psi$（分析器）：诊断失败轨迹，输出标签 $\phi$ 和分析 $a$。
  - $\sigma$（检索器）：根据标签 $\phi$ 检索同分支和跨分支的灵感节点。
  - $\alpha$（分配器）：每轮决定子代预算 $K \in [1, K_{\max}]$。
  - $\pi$（提议器）：基于失败分析、检索灵感生成具体编辑提案 $\delta$。
  - $\varepsilon$（进化器）：将编辑写入磁盘并验证一致性。
- **快循环（Fast Loop）**：每轮更新任务技能 $s$，在训练批上找到最差样本驱动管道，产生 $K$ 个子代，评估其在验证集上的效用增益 $\Delta U$。
- **慢循环（Slow Loop）**：每 $H$ 轮快循环执行一次，聚合最近 $H$ 个后代的平均元生产力 $\hat{P}(m|s)$，构造元失败轨迹 $f_m$，使用相同五智能体管道对元技能文件 $m$ 进行重写（同时编辑所有五个组件文件），实现递归自改进。

### 2.2 前沿选择（Frontier Selection）

从候选池（存档）中选下一父代节点时，评分函数为：
$$
v^* = \arg\max_{v \in \mathcal{F}} \left[ \eta_1 U_v + \eta_2 \hat{P}_v + \eta_3 N_v \right]
$$
其中 $U_v$ 是任务效用，$\hat{P}_v$ 是元生产力（子代平均提升），$N_v = 1/(1+\text{times\_selected}_v)$ 是访问冷却项。三个项分别防止：仅择优停滞、忽略改进策略质量、一条线独占预算。

### 2.3 递归自改进的封闭性

元技能的每个组件本身是**Markdown技能文件**，与任务技能格式相同。因此同一五智能体管道可递归应用于改进元技能本身，无需额外模型或训练。这是“有界的单层递归”。

## 6. 主要结论与发现

1. **递归自改进有效**：在保留测试集上，MetaSkill-Evolve相比No-Skill提升OfficeQA +23.54点、SealQA +16.09点、ALFWorld +1.92点。
2. **元技能慢循环贡献显著**：相比Single-Level Evolution（固定改进过程）额外提升OfficeQA +6.38、SealQA +8.05、ALFWorld +1.92。
3. **每类元技能组件均贡献**：消融实验中任一组件缺失都有性能下降，且最重要组件因领域而异（OfficeQA中$\alpha$最关键，SealQA中$\pi$最关键）。
4. **地平线 $H=2$ 最优**：更频繁的元更新带来更好性能，OfficeQA对元技能老化最敏感。
5. **跨分支检索**在饱和基准（ALFWorld）上贡献了全部提升；在QA基准上也一致有益。

## 7. 优点

- **递归自改进的创新设计**：将改进过程本身作为可进化对象，实现“学习如何学习”，无需额外模型或训练。
- **轻量且通用**：所有五智能体共享同一冻结骨干，改进仅通过编写技能文件实现，不增加模型容量。
- **元生产力指标 $\hat{P}$** 提供质量多样性信号，避免仅依赖效用 $U$ 导致的搜索停滞。
- **SQLite持久化进化图**支持跨分支检索和完整溯源，避免固定大小缓冲区的局限。
- **充分的消融与敏感性分析**验证了每个组件的必要性及超参数鲁棒性。

## 8. 不足与局限

- **基准覆盖有限**：仅在三个精心策划的基准上测试，未验证在更多样化、开放、长周期真实世界任务上的表现，噪声反馈场景下性能未知。
- **五智能体管道本身固定**：虽然元技能参数可进化，但管道角色和连接方式（分析→检索→分配→提议→进化）在设计时固定，未实现管道结构自身的递归进化。
- **元更新地平线 $H$ 固定**：论文中设为固定值2，未探索自适应地平线机制（例如根据任务复杂性动态调整）。
- **跨分支检索可能引入噪声**：对ALFWorld这种近饱和基准，过度激进的跨分支检索反而有害。
- **计算成本未量化**：虽然骨干冻结，但多次LLM调用（每轮快循环调用五智能体，加上慢循环）在实际部署中可能成本较高，论文未提供详细成本分析。
