## 摘要
长期大规模语言模型系统中的智能体自我进化主要是程序性的：有用的经验不仅仅是存储的信息，而是可重用的搜索、调试和验证程序。然而，当前的评估并未孤立这种形式的迁移。智能体基准测试测试单次任务解决；记忆基准测试针对信息保留而非程序重用。我们引入了EvoAgentBench，这是一个通过能力引导迁移实现智能体自我进化的基准测试，涵盖四个智能体领域：网络研究、算法推理、软件工程和知识工作。EvoAgentBench从智能体执行中提取基于轨迹的能力，将其规范化操作为操作单元，并构建特定领域的能力图，连接具有程序重叠的任务。通过设计，每个测试任务都有经过验证的训练侧能力支持。在528/267的训练/测试划分、两种脚手架和三种骨干网络上，精心策划的能力内容在不同模型系列之间可靠迁移，但目前没有自动方法能在所有设置中维持正向增益。EvoAgentBench将自我进化评估从整体准确性比较转向对经验编码、路由和吸收的细粒度诊断。该基准测试公开于https://huggingface.co/datasets/EverMind-AI/EvoAgentBench。

## Abstract
Agent self-evolution in long-horizon LLM systems is largely procedural: useful experience is not merely stored information, but reusable procedures for searching, debugging, and verification. Yet current evaluations do not isolate this form of transfer. Agent benchmarks test single-episode task solving; memory benchmarks target information retention rather than procedural reuse. We introduce EvoAgentBench, a benchmark for agent self-evolution via Ability-guided transfer across four agentic domains: web research, algorithmic reasoning, software engineering, and knowledge work. EvoAgentBench extracts trace-grounded Abilities from agent executions, canonicalizes them into operational units, and builds domain-specific Ability Graphs linking tasks that share procedural overlap. By design, every test task is backed by verified training-side Ability support. Across a 528/267 train/test split, two scaffolds, and three backbones, curated Ability content transfers reliably across model families, but no current automatic method sustains positive gain in all settings. EvoAgentBench shifts self-evolution evaluation from aggregate accuracy comparison to fine-grained diagnosis of experience encoding, routing, and uptake. The benchmark is publicly available at https://huggingface.co/datasets/EverMind-AI/EvoAgentBench.

### 1. 核心问题与整体含义（研究动机和背景）

- **现有评估盲区**：当前智能体基准（如 BrowseComp、SWE-Bench）只测试单次任务解决能力；记忆基准（如 LongMemEval）只评估信息存储与检索，均未测量智能体从过往执行中提取**可重用过程性技能**并迁移到相关但不同任务的能力。
- **关键缺口**：智能体的“自我进化”应包含对搜索、调试、验证等程序化经验的编码与复用，而非仅靠长上下文或记忆召回。没有基准能系统地、细粒度地诊断这种“轨迹→程序”的迁移效果。
- **本文目标**：提出 EvoAgentBench，通过**能力图**（Ability Graph）控制训练侧对测试任务的技能支持，从而测量智能体自我进化方法在过程性迁移上的表现。

### 2. 论文提出的方法论

#### 核心思想
- 定义“能力”（Ability）为从执行轨迹中抽取的、带触发条件和可操作性的程序化单元（如搜索策略、调试流程、验证步骤），而非领域标签或主题。
- 基于能力图构建训练/测试拆分，保证每个测试任务至少与一个训练任务共享一种可迁移的能力，从而在等价的设置下只比较不同方法在经验编码、路由和吸收上的差异。

#### 关键技术细节（三阶段流水线）
1. **多骨干轨迹收集**：对每个任务，使用三个构造骨干（Kimi-K2.5, GLM-5.1, DeepSeek-V3.2）和两个脚手架（OpenClaw, Nanobot）收集无技能的执行轨迹，形成多视角证据池。
2. **轨迹接地的能力提取**：
   - 提取器 Eϕ（使用 Claude Sonnet 4.6）联合任务查询、真实目标、多骨干轨迹，输出`(触发条件, 过程描述, 证据, 边界, 角色)`五元组（角色分 Method/Guard/Workflow）。
   - 提取得到 7326 张原始能力卡，覆盖 2516 个任务。
3. **能力规范化与能力图构建**：
   - 通过嵌入相似度（cosine>阈值）做粗选候选对，再经三个 LLM 独立判断 + 领域专家审核，合并操作等价的能力。
   - 最终得到 170 个规范能力单元，构建无向能力图：边表示两个任务共享至少一个能力单元。
   - 用 Louvain 社区检测划分能力社区，在社区内部进行约束拆分，保证每个测试任务至少有一个能力单元有训练侧任务支持（**能力感知的迁移拆分**）。
4. **诊断参考：Anchor Skill**：使用构造骨干在训练迹上提取的能力文本，由策展侧能力标签路由到测试任务，作为性能上限参考。

### 5. 实验数量与充分性

- **实验规模**：共 4 领域 × 2 脚手架 × 3 骨干 × 5 方法（含 Vanilla）= 120 个条件-领域组合，每个 3 次重复，约 360 次完整运行。加上消融（成本分析、按解决/未解决分解），数量可接受。
- **充分性**：
  - **积极**：覆盖多领域、多骨架、多骨干，且分别报告每个 cell 的 ∆，可暴露特定环境下的负迁移；成本分析补全了使用视角。
  - **不足**：每个领域测试集只有 56–86 个任务，对于细粒度能力级分析样本量偏小。自动方法仅涵盖三类（检索案例、检索记忆、单提示进化），未包括近期 Skill 库学习方法（如 SkillPro, SkillOS）。另外仅使用文本域，缺少多模态或具身智能体评估。
- **公平性**：框架强制任务、工具、超时、评分合同在方法间相同，仅进化状态不同，比较合理。Anchor 作为诊断参考，其构建骨干与评估骨干分离，避免了污染。

### 6. 论文的主要结论与发现

1. **能力内容跨模型可迁移**：Anchor Skill 在所有 24 个 cell 上均取得正面 ∆（平均 +5.8 ~ +10.5），表明训练侧能力图提供的信息足以跨模型系列提高性能。
2. **自动方法不稳定，常出现负迁移**：每个自动方法至少有一个 cell 是负 ∆。例如 Memento 在 Nanobot/Qwen3.5-27B/SWE 上骤降 36.3 个百分点；GEPA 在 OpenClaw/Qwen3.5-27B/KW 上下降 12.3 个百分点。系统瓶颈在方法侧的经验编码和路由机制，而非能力本身。
3. **进化并不统一降低成本**：整体上，Anchor 在 4/6 设置下节省回合数；自动方法通常增加成本（尤其 Gemma-4-31B），且即使对已解任务，某些方法仍增加回合数（如 Memento +39.4%）。成本与准确率脱离，需联合报告。
4. **单一骨干/脚手架评估不可靠**：方法排名随骨干和脚手架变化，多设置评估对稳健性对比至关重要。

### 7. 优点

- **概念创新**：首次明确定义“过程性能力迁移”的评估问题，并引入能力图原理（ability-supported yet instance-disjoint）保证测试任务有可迁移支持。
- **细粒度诊断**：将自我进化评估从聚合准确率拆解为“经验编码、路由、吸收”环节，通过 Anchor 诊断参考定位失败元凶。
- **多领域、多骨架、多骨干覆盖**：4 个代表性长程智能体领域 × 2 种不同复杂度脚手架 × 3 个开放模型（含不同尺寸），结果具有较好的泛化性。
- **成本-性能联合分析**：揭示成本并不是精度的指示符，避免单一指标误导。
- **开放数据与代码**：基准数据集公开，有利于社区复现和进一步研究。

### 8. 不足与局限

- **领域覆盖有限**：仅文本型长程智能体任务（搜索、代码、软件工程、知识工作），未涉及多模态、具身或对话式智能体。
- **构造骨干限制**：能力提取仅使用一个 LLM（Claude Sonnet 4.6）和三个构造骨干（Kimi, GLM, DeepSeek），可能遗漏其他模型提取的能力或引入特定偏差；规范化的审核依赖人力和有限专家，扩展性有待验证。
- **测试规模建议细化**：每个领域测试集仅 56–86 个任务，较难支撑能力级统计检验（论文也承认需要更大规模）。
- **数据污染风险**：评估骨干可能在预训练时见过源基准数据；虽然方法间公平，但绝对分数可能有偏高。
- **拆分偏向**：测试任务优先选择构造骨干留有改善空间的样例（headroom），因此 ∆ 幅度不代表在全任务随机抽样下的预期收益，仅反映在该特定支持性子集上的表现。
