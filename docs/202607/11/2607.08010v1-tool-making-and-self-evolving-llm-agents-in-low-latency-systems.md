## 摘要
生产环境中的大语言模型智能体常常因为对每个请求的相同程序步骤重新生成代码而浪费延迟和可靠性。我们用一个智能体工具制造流水线替代了这种推理时的编码循环，该流水线在部署前将重复的标准操作步骤编译为经过验证、有版本控制的工具。工具制造者在收集执行轨迹、观察后端模式与数值、生成候选工具并根据标注案例进行修复的过程中，将综合过程锚定于实时环境。运行时，生产智能体直接调用这些工具，仅在必要时回退到代码生成。我们在一个履约中心警报分类系统中部署了该方法，该系统的智能体针对44个节点的标准操作程序在异构指标后端上诊断警报。在生产中，工具调用将p50延迟降低了42%。在1500个历史警报上，通过抑制重复步骤的逐次运行方差，端到端错误率降低了最高53%。由于工具返回紧凑的结构化判决，它们还实现了更简单的直接调用架构，在受控消融实验中将p50延迟进一步降低了62%。版本化工具还改善了可审计性，并暴露了规格缺口和上游数据漂移。我们的结果表明，自进化智能体可以使工业大语言模型系统更快、更可靠、更易于操作。

## Abstract
Production LLM agents often waste latency and reliability by regenerating code for the same procedural steps on every request. We replace this inference-time coding loop with an agentic tool-making pipeline that compiles repeated SOP steps into validated, versioned tools before deployment. The tool-maker grounds synthesis in the live environment as it collects execution traces, observes backend schemas and values, generates candidate tools, and repairs them against labeled cases. At runtime, the production agent calls these tools directly and falls back to code generation only when needed. We deploy the approach in a Fulfillment Center alarm-triage system, where an agent diagnoses alarms against a 44-node SOP over heterogeneous metric backends. In production, tool calls reduce p50 latency by 42%. On 1,500 historical alarms, they reduce end-to-end error rate by up to 53% by suppressing run-to-run variance in repeated steps. Because tools return compact structured verdicts, they also enable a simpler direct-call architecture, reducing p50 latency by a further 62% in a controlled ablation. Versioned tools also improve auditability and expose specification gaps and upstream data drift. Our results show that self-evolving agents can make industrial LLM systems faster, more reliable, and easier to operate.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：生产环境中，LLM 智能体在每次请求时都为相同的标准操作步骤重新生成代码，导致高延迟、低可靠性和逐次运行方差。这种“推理时代码生成”模式在面对重复性任务时效率低下。
- **意义**：本文提出通过“自进化智能体”范式，让智能体在部署前将重复的 SOP 步骤编译为经过验证、版本化的工具，从而在运行时直接调用，避免重复编码，大幅提升工业系统的速度、可靠性及可审计性。

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：从“每次推理都写代码”转变为“一次构建、多次调用”。在离线阶段，每个 SOP 节点被编译为一个确定性的工具；在线阶段，智能体直接调用这些工具，仅在工具不可用或失败时回退到代码生成。
- **关键技术细节**：
  - **工具制造流水线**（图1）：
    - **数据收集子智能体**：对每个 SOP 节点，在实时环境（MCP）上运行3个标注样本，记录执行轨迹（查询代码、MCP 响应、观察到的模式、评分结果）。
    - **工具制造 LLM**：基于 SOP 文本、决策树位置和数据收集轨迹，生成候选工具。
    - **测试-修复循环**：将候选工具在完整训练集上测试，失败案例由反射 LLM 诊断，然后工具制造 LLM 重写，最多3轮。
  - **工具签名**：统一接口 (warehouse, timestamp, context) → (verdict: true/false/no data, 支持细节)。
  - **智能体微调**：通过蒸馏大教师模型（GLM-4.7）到小学生模型（Qwen3 32B / GLM-4.5-Air），使用 LoRA 微调。教师轨迹在不同工具可用性下生成（75% 可用，25% 随机禁用部分工具，25% 完全无工具），保证学生模型能稳健回退。

## 6. 主要结论与发现

- **延迟**：工具调用使 p50 延迟降低 42%（sub-agent 架构）；直接调用架构进一步降低 62%（GLM-4.5-Air 下受控消融）。

- **可审计性与运维**：版本化工具使输入输出稳定，日志可暴露上游数据漂移和环境不一致（3个具体案例）。
- **工具可迁移**：一次构建的库可被不同智能体模型使用；参数化工具可跨部署复用。

## 7. 优点

- **方法新颖实用**：将推理时代码生成转化为离线工具编译，兼具灵活性与效率。
- **强环境接地**：数据收集轨迹提供了 SOP 文本缺失的执行细节，修复循环结合真实标注，显著提升工具正确率。
- **严密实验设计**：大量消融、统计检验、生产部署验证，结论坚实。
- **低标签依赖性**：仅需少量标注案例（N=5 即接近天花板），甚至无标签配置也能达到 91.6% pass@1（距天花板 3pp）。
- **运维友好**：版本化工具支持自动化漂移检测、回滚无需重新训练；决策树架构允许细粒度监控。
- **实际部署证据**：在真实物流中心系统上运行5,000+报警，非仅基准测试。

## 8. 不足与局限

- **应用领域单一**：仅部署于一个特定应用（出库码头告警分类），对不同 SOP 类型（自由式运行手册等）的泛化性未验证。
- **人类监督依然必要**：规格澄清需要人工；工具生成后的审查仍可能有遗漏，全自动化尚未实现。
- **标注依赖**：虽然有标签高效，但创建新节点仍需少量标注案例；无标签变体尚有 3pp 差距。
- **模型依赖**：工具制造依赖较强模型（GLM-4.7/Opus），较弱模型（Qwen3 235B/GLM-4.7 Flash）提升较小甚至负收益。
- **潜在偏差**：实验在单一公司基础设施上完成，特定 MCP 接口和图谱可能影响结果普遍性。
- **未探索大规模迭代**：工具库动态扩展时的版本管理和冲突解决未被讨论。
