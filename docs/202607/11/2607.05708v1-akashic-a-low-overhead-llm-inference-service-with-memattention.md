## 摘要
近期的基于LLM的智能体系统在多轮交互、工具调用和跨会话工作流中持续积累上下文。为每个请求重放完整历史很快变得不切实际：长上下文增加预填充成本，可能超出上下文限制，并且常常将任务相关证据埋没在不相关内容中，降低了服务效率和输出质量。我们提出Akashic，一种围绕MemAttention构建的低开销内存系统，它将上下文组织成有界块并对跨块的语义关系进行建模，无需重复重写完整历史即可保留跨块证据。Akashic进一步应用硬件-软件协同设计的内存布局，将可能共同检索的块放在一起，减少检索碎片化和I/O开销。在四种代表性工作负载和三种模型大小上，与强大的先前内存基线相比，Akashic将任务准确率提升多达10.2个百分点，吞吐量提升多达1.21倍，可持续请求率提升多达1.88倍。

## Abstract
Recent LLM-based agent systems continuously accumulate context across multi-turn interactions, tool invocations, and cross-session workflows. Replaying the full history for every request quickly becomes impractical: long contexts increase prefill cost, may exceed context limits, and often bury task-relevant evidence in irrelevant content, degrading both serving efficiency and output quality. We propose Akashic, a low-overhead memory system built around MemAttention, which organizes context into bounded chunks and models semantic relationships across chunks, preserving cross-chunk evidence without repeatedly rewriting the full history. Akashic further applies hardware-software co-designed memory placement to co-locate likely co-retrieved chunks, reducing retrieval fragmentation and I/O overhead. Across four representative workloads and three model sizes, Akashic improves task accuracy by up to 10.2 points, throughput by up to 1.21x, and sustainable request rate by up to 1.88x over strong prior memory baselines.

# Akashic：一种基于MemAttention的低开销LLM推理服务

## 1. 核心问题与整体含义（研究动机与背景）
- **问题**：LLM智能体系统在多轮交互、工具调用和跨会话工作流中持续累积上下文，导致预填充成本增加、可能超出模型上下文窗口限制，且关键证据被无关内容淹没，降低服务效率和输出质量。
- **现有局限**：现有记忆系统（如Mem0、MemGAS、MemGPT、RMM）要么采用全局压缩（更新成本随历史增长、引入噪声），要么进行独立段级压缩（丢失跨段依赖）；并且存储侧缺乏物理布局优化，导致语义相关与物理分散的“局部性鸿沟”。
- **目标**：设计一种低开销的记忆系统，在保持高任务质量的同时提升推理服务效率。

## 2. 方法论：核心思想、关键技术细节
### 2.1 核心思想
- **MemAttention**：将上下文组织成有界块（chunk），进行增量式块级维护，并通过跨块语义建模保留跨块证据，避免重复重写完整历史。
- **硬件-软件协同的内存管理**：将可能协同检索的块物理上放置在一起，减少I/O碎片和检索争用。

### 2.2 关键技术细节
#### 2.2.1 MemAttention 算法（Algorithm 1）
- **输入**：新轮次内容、活跃块、块表、门限τ_c、top-p=5。
- **流程**：
  1. 将新轮次追加到活跃块。
  2. 当活跃块长度≥τ_c时触发压缩：生成紧凑表示并提取元数据关键词。
  3. 收集同一命名空间下已压缩块的元数据，用LLM进行语义匹配，选出top-p个相关先前块。
  4. 联合压缩（JointCompact）以融合新块与先前块，同时更新或删除过期记忆。
  5. 写回新块，启动新活跃块。
  6. 推理时：将当前查询与候选块元数据送入LLM，选出top-p相关块，按时间顺序拼接后作为提示前缀。

#### 2.2.2 内存管理器（Algorithm 2）
- **关联感知重定位**：模型根据元数据推断块间亲和度，将频繁共同访问的块重写到新块（out-of-place），标记旧副本无效。
- **后台压缩（垃圾回收）**：当块无效比率≥0.75时，提取有效块，按(user, session)分区，再按模型推断的亲和度分组，打包到新块，释放旧块。
- **目标**：减少跨块推理时的磁盘读次数，降低检索延迟，同时保持适度的空间放大和写放大。

## 7. 优点
- **方法创新**：
  - 将记忆维护与物理布局联合优化，首次系统性地解决LLM智能体记忆的语义-物理鸿沟。
  - 使用同一LLM驱动的语义匹配机制同时用于记忆维护和查询检索，统一关联概念。
  - 采用日志结构合并树（LSM-tree）式更新策略，实现可控开销下的增量压缩。
- **实验全面**：
  - 覆盖多种信息密度模式（高密度、突发性、长期对话），验证鲁棒性。
  - 消融实验精准量化各组件贡献。
  - 参数敏感性分析表明系统对超参数不敏感（默认值接近最优）。
- **实现轻量**：仅扩展vLLM控制路径，不修改GPU解码引擎，易于集成。

  - 未测试跨会话、多用户混合负载下存储器管理的扩展性（仅单用户会话内评估）。

- **偏差风险**：所有基线均使用相同LLM进行记忆操作，但Akashic利用同一模型进行语义匹配，可能引入与方法无关的模型偏差；论文未采用不同LLM作为验证。
