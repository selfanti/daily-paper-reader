## 摘要
设计有效的Lean证明代理是形式化数学推理中的核心挑战。除了构建更强的证明器外，近期工作强调了围绕Lean的工作流程：代理如何分解证明义务、使用工具和编译器反馈、诊断失败、修复证明以及维护结构化的证明上下文。受代码级自进化代理的启发，我们研究此类工作流程是否可以进化而非手动设计。我们提出了一种自进化的Lean证明代理，其中一个小型固定、可信的运行时封装了一个完全可变的工作区：包括证明工作流程、提示和工具。与大多数针对固定外部基准进行优化的自进化系统不同，我们的系统协同进化代理及其基准。在代际之间，最高得分的代理（冠军）通过一种掌握限制的课程更新来修订活跃的任务分布，仅在当前级别掌握后才引入更难的证明义务，并且通过单锚定重校准在更新后的基准上重新运行冠军，以在难度增加时保持分数可比性。所有进化都保持在Lean基础的验证循环内：无论代理如何重写自身，只有当其行为在可信快照下产生Lean验证的证明时才算成功，并且每次尝试必须输出机器可读的、基于Lean的证明上下文，其表示形式可以进化，但基础性得以强制执行。我们运行了协同进化轨迹和固定基准基线，共15个活跃世代，并在一个保留的miniF2F测试集上进行对比。最佳协同进化代理达到了45.1%的保留测试解决率，而种子代理为12.7%，最佳固定基准代理为32.0%，这表明验证器基础的自进化可以在协同进化基准下改进Lean证明工作流程。

## Abstract
Designing effective Lean proof agents is a central challenge in formal mathematical reasoning. Beyond building stronger provers, recent work emphasizes the workflow around Lean: how an agent decomposes proof obligations, uses tools and compiler feedback, diagnoses failures, repairs proofs, and maintains structured proof context. Motivated by code-level self-evolving agents, we study whether such workflows can be evolved rather than hand-designed. We present a self-evolving Lean proof agent in which a small fixed, trusted runtime wraps a fully mutable workspace: the proof workflow, prompts, and tools. Unlike most self-evolving systems, which optimize against a fixed external benchmark, our system coevolves the agent and its benchmark. Between generations, the highest-scoring agent (the champion) revises the active task distribution through a mastery-throttled curriculum update that introduces harder proof obligations only after the current level is mastered, and a single-anchor recalibration re-runs the champion on the updated benchmark to keep scores comparable as difficulty rises. All evolution stays inside a Lean-grounded verification loop: however the agent rewrites itself, a success counts only when its behavior yields Lean-verified proofs under a trusted snapshot, and each attempt must emit a machine-readable, Lean-grounded proof context whose representation may evolve but whose groundedness is enforced. We run the coevolving trajectory and a fixed-benchmark baseline for 15 active generations and compare them on a held-out miniF2F test split. The best coevolving agent reaches a 45.1% held-out solve rate, versus 12.7% for the seed and 32.0% for the best fixed-benchmark agent, showing that verifier-grounded self-evolution can improve Lean proof workflows under a coevolving benchmark.

- **研究动机**：在形式化数学推理中，Lean 证明代理的设计日益重要。现有工作分为两类：一类构建更强的专用证明器（如搜索策略、模型），另一类强调工作流程（如分解证明义务、利用编译器反馈、修复证明）。后一类（如 LEAP、Goedel-Architect）虽取得高性能（Goedel-Architect 在 miniF2F 上达 99.2%），但工作流程完全由人工设计、固定不变。

- **整体含义**：论文表明，通过将 Lean 作为固定、可信的验证器，允许代理围绕验证器重写自身代码（工作流、提示、工具），并让基准根据掌握程度自动调整难度（共同进化），可以显著提升证明能力（从 12.7% 到 45.1% 的 held-out 解决率）。这为自进化推理系统提供了可验证的范式。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：将系统分为**固定可信运行时**和**可变工作空间**。固定运行时负责 Lean 验证、评测、基准管理等不可篡改的部分；可变工作空间包含证明工作流、提示和工具，是进化的对象。进化的唯一奖励来源是 Lean 验证的结果（而非代理自报的成功）。同时，基准也随冠军代理的表现而动态调整难度，称为“验证器基础基准共同进化”。
- **关键技术细节**：
  - **自进化机制**：借鉴 Darwin Gödel Machine（DGM）和 Hyperagents，代理通过代码级改写（mutation）自身工作流。每个变异后代先通过语法和烟雾测试，然后被评估，成功以 Lean 验证为准。
  - **证明上下文合约**：每个尝试必须输出 {ok, proof_body, proof_context} 三元组。proof_context 可自由表示（图、树、列表等），但必须满足 groundness 条件（有根节点、至少一个数学声明、依赖指向已知项、标记已解决的必须被 Lean 验证等）。固定运行时独立重新验证所有证据，防御 agent 欺骗。
  - **共同进化基准**：候选问题池分为三个难度等级 L1（单战术引理）、L2（miniF2F valid 子集）、L3（PutnamBench）。初始活跃基准包含 76 个任务，难度按 L1:L2:L3 = 27:46:3 分配。基准更新规则：
    1. 冠军原始解决率 ≥ γ=0.30 时才触发更新。
    2. 那些被所有本轮代理解决的旧任务可退役（最多 κ=6 个）。
    3. 替换规则：若冠军在该级别解决率 ≥ τ=0.70，则替换为更高级任务；否则同级别横向替换；优先选择弱域任务。
    4. **单锚定重校准**：更新后冠军在新区块链上重新评估，计算难度系数 c_{t+1} = c_t * r_t / max(r_{t+1}, ε)，使得冠军归一化分数保持不变，从而跨世代分数可比。
  - **父代选择**：在世代内，根据归一化分数 q = r × c 和子代数量折扣 w = max(q,0) * 1/(1+n_c) 采样父代，允许流式选择（早期强子代可立即成为父母）。

## 6. 论文的主要结论与发现
- **主要结论**：验证器基础的自我进化（结合共同进化基准）可以显著改进 Lean 证明代理，使 held-out miniF2F 解决率从 12.7%（种子）提升至 45.1%（第 15 代冠军），而固定基准最佳仅达 32.0%。
- **关键发现**：
  - **胜出工作流是修复中心而非分解中心**：高绩效代理主要采用简洁的 Lean 错误修复循环（生成完整证明 → 检查错误 → 重试），而非基于蓝图的分解工作流。
  - **工具演化**：绝大多数进化工具旨在检测和拒绝幻觉的 Lean 名称（如 #check 探针、缓存型引理检查器），使修复更安全。
  - **证明上下文的局限性**：虽然上下文变得可检查，但并未成长为深度验证的证明图；大多数节点是“have”型浅层支持，未形成可复用的 DAG。

  - **基准共同进化的必要性**：固定基准在早期饱和后缺乏梯度，进化偏向局部修复；共同进化基准迫使代理不断应对更高难度任务。
